# Skill 02: 实施修复

> **触发条件**：Phase 02 指示（入口决策 Q2 为「否」）  
> **目标**：对「已确认」任务实施修复，确保验证测试通过且全量回归无新增失败。

---

## 输入

- Phase 01 的验证结果与验证测试（仅处理「已确认」任务）
- `docs/risk_tasks/task_list.md`
- `docs/remediation/remediation_log.md`（含每条任务的验证测试路径）
- 按需的 `docs/codearch/modules/<module_name>.md`、`docs/codearch/build_and_tests.md`
- 仓库源码

---

## 输出

| 产出 | 路径 | 必须 |
|------|------|------|
| 源码修复 | 仓库源码（直接修改，当前分支） | ✓（对已确认任务） |
| 验证测试通过 | Phase 01 的验证测试 | ✓（修复后验证测试须通过） |
| 全量回归通过 | 工程测试套件 | ✓（修复不得引入新的测试失败） |

---

## 核心任务

### 任务 1: 对每条已确认任务实施修复并验证

1. 根据 Phase 01 的验证结果，筛选出「已确认」任务。
2. **每条已确认任务**执行以下修复-验证循环：
   - **a. 定位与修复**：根据「位置」「简要描述」「风险类型」「推理依据」定位代码，实施修复（直接修改源码，在当前分支上）。
   - **b. 运行验证测试**：运行 Phase 01 为该任务编写的验证测试。修复后，该测试应当**通过**（即 BUG 不再可触发）。
   - **c. 运行全量回归测试**：按 `docs/codearch/build_and_tests.md` 运行完整测试套件，确认修复未引入新的测试失败。
   - **d. 迭代**：若验证测试未通过或回归测试出现新失败，调整修复方案并重复 b-c 步骤，直至验证测试通过且回归无新增失败。
   - **e. 记录**：在 `docs/remediation/remediation_log.md` 中更新该任务的修复信息：
     - 修复描述（做了什么改动）
     - 修复前验证测试结果（FAIL — 因为 BUG 存在）
     - 修复后验证测试结果（PASS — BUG 已修复）
     - 全量回归测试结果（PASS / 有 N 个预已存在的失败）
     - 修复涉及的文件列表
3. 若本轮回溯无「已确认」任务，在 remediation_log 中注明「本轮回溯无可修复任务」即可。

### 任务 2: 验收自检

- 确认所有已确认任务均已实施修复（源码已变更）；
- 确认每条已修复任务的验证测试在修复后通过；
- 确认全量回归测试无新增失败；
- 确认 remediation_log 中已更新修复信息。

---

## 验收

与 [Phase 02](3-bug-remediation/phases/02-fix.md) 阶段验收标准一致：已确认任务均已修复、可检查通过。

---

## 状态跳转

| 验收结果 | 下一步 |

| 未通过 | 补全修复后重新验收 |
