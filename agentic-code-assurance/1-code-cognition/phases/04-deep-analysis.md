# Phase 04: 深度分析

> **阶段代号**: P4  
> **入口条件**: P3 收敛评估通过，或已达迭代上限  
> **预计耗时**: 小型项目 1–2 小时 · 中型 2–4 小时 · 大型 4–8 小时

---

## 目标

将所有模块（父模块与叶子模块）的报告从 L1 升级为 L2，执行全深度分析。

L1 报告是迭代期间的轻量中间产物；P4 在此基础上补充完整的架构细节，产出最终 L2 报告，供下游工作流（风险分析、代码审查、重构、写测试等）消费。

---

## P4 新增内容

在 L1 报告已有字段（模块名、路径、代码指标、职责、依赖、复杂度评级）的基础上，P4 为每个模块补充以下 L2 专属字段：

| [P4 新增] 字段 | 说明 |
|---|---|
| **关键设计要点** | 3–5 条要点：主要入口函数/类、核心数据结构、线程/并发假设、错误处理策略、扩展点等 |
| **代码特征** | 内存管理（RAII/智能指针/手动/混合）、并发模型（单线程/多线程+同步机制）、I/O 操作（文件/网络）、外部数据处理（是否处理外部输入+来源）、错误处理（返回错误码/异常/混合） |
| **关键代码位置索引** | 主要入口点（函数/类、位置、说明）、外部数据入口（含首道校验位置）、错误/异常路径（触发条件、清理逻辑位置、是否完整） |
| **关键数据流路径** | 模块内关键数据从入口到出口的流转路径，每条含起点、经过、终点、涉及的所有权转移 |
| **生命周期与所有权模型** | 关键对象/资源的创建位置、销毁位置、所有权模型、转移点（涉及手动内存管理或跨函数/跨模块生命周期时必填；无动态分配模块标注「无需记录」） |
| **并发不变量** | 受保护数据/结构、保护锁/机制、锁获取顺序约定、线程上下文假设（仅多线程模块必填；非多线程模块标注「本模块为单线程模型」） |
| **与其它模块的关系** | 结构化表格：上游依赖（模块名、依赖类型、跨模块所有权转移、说明）+ 下游依赖（同结构）。依赖类型枚举：调用/数据传递/事件通知/配置读取/继承 |
| **信息来源** | 分析所参考的代码文件、文档、注释等 |
| **使用示例** | 至少一个典型使用示例 |
| **验证状态** | 仅 **高/极高** 复杂度模块需填写 |
| **子模块索引** | 仅 **父模块** 需填写：深入分析日期/轮次、拆分依据、子模块列表（名称、路径、复杂度、报告链接） |

## 验收标准

### 报告层级

- [ ] 所有模块报告已升级为 L2（检查每份报告中的 `报告层级: L2` 标记）

### L2 报告内容完整性

- [ ] 每份 L2 报告包含 [output_structure](../definitions/output_structure.md) 要求的全部必填章节
- [ ] **代码特征**：内存管理、并发模型、I/O 操作、外部数据处理、错误处理五项均已填写
- [ ] **关键代码位置索引**：主要入口点、外部数据入口（若有）、错误/异常路径均已列出
- [ ] **关键数据流路径**：每个模块至少包含一条数据流路径
- [ ] **生命周期与所有权模型**：涉及手动内存管理的模块已填写；无动态分配模块已标注「无需记录」
- [ ] **并发不变量**：多线程模块已填写；非多线程模块已标注「本模块为单线程模型」
- [ ] **与其它模块的关系**：结构化表格（上游/下游）已填写，依赖类型与总体报告依赖图一致
- [ ] **使用示例**：每个模块至少包含一个使用示例
- [ ] **验证状态**：高/极高复杂度模块已填写验证状态

### 父模块与总体报告

- [ ] 父模块已添加「子模块索引」章节
- [ ] 总体报告的「技术特征统计」已更新

### 验证命令

```bash
# 检查所有报告是否为 L2
for f in docs/codearch/modules/*.md; do
  grep -q "L2" "$f" && echo "OK $f" || echo "CHECK $f (may need L2 upgrade)"
done

# 检查关键章节是否存在
for f in docs/codearch/modules/*.md; do
  grep -q "代码特征" "$f" && grep -q "关键数据流" "$f" && echo "OK $f" || echo "MISSING sections: $f"
done
```
