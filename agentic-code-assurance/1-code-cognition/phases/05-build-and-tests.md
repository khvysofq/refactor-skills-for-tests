# Phase 05: 编译与测试门禁

> **阶段代号**: P5  
> **入口条件**: P4 深度分析已完成  
> **预计耗时**: 小型项目 20–40 分钟 · 中型 40–90 分钟 · 大型 1–2 小时

---

## 目标

梳理编译系统、单元测试体系及测试运行方式，并文档化；实际执行构建与测试，验证环境可用性。

> **硬性门禁**：本阶段是整个工作流的硬性门禁。若构建失败或测试无法运行，整个工作流中止。

---

## 执行

加载并执行 [Skill 05 — 编译与测试](../skills/skill-05-build-tests.md)。

---

## 验收标准

### 必须满足

- [ ] **构建与测试文档存在**

  ```bash
  [ -f docs/codearch/build_and_tests.md ] && echo "PASS" || echo "FAIL"
  ```

- [ ] **文档包含**：
  - [ ] 编译系统说明（类型、主入口、至少一条构建命令）
  - [ ] 单元测试说明（有则写框架与运行方式，无则写明「当前无单元测试体系」）
  - [ ] 运行测试的方式（命令或说明「无测试可运行」）

- [ ] **构建命令已实际执行且返回 exit code 0**（构建成功）
- [ ] **测试套件已实际运行**（或已确认无测试体系并触发中止）
- [ ] **文档包含「环境验证状态」章节**（含 build_success、tests_runnable、test_result、blocking_issues）

### 可选

- [ ] 包含不同配置（Debug/Release）或平台的构建说明
- [ ] 包含测试过滤或按 target 运行的说明

---

## 环境门禁（硬性检查）

Skill 05 完成后，根据「环境验证状态」章节做硬性判定：

| 条件 | 判定 |
|---|---|
| build_success=true 且 tests_runnable=true | **通过**：继续后续流程 |
| build_success=false | **中止工作流**：输出明确信息告知使用者当前环境无法编译工程，列出阻塞问题，要求配置好编译环境后重新执行 |
| tests_runnable=false | **中止工作流**：输出明确信息告知使用者单元测试无法运行，列出原因，要求确保测试体系可运行后重新执行 |

**中止时的输出格式**：

```
⛔ 工作流中止：编译/测试环境未就绪

问题描述：<具体问题>
阻塞原因：<原因列表>

请先解决上述问题，确保工程可编译且测试可运行，然后重新执行工作流。
```
