# Skill 02: 证据驱动复杂度评级

> **触发**: Phase 02 指示  
> **目标**: 为每个模块执行证据驱动的复杂度评级  
> **预计耗时**: 30 分钟 – 4 小时（取决于项目规模）

---

## 输入

| 输入 | 来源 | 说明 |
|---|---|---|
| L1 模块报告 | P1 产出 | 包含模块路径和代码指标 |
| 源代码访问 | 项目源码树 | 用于采集模块级客观指标 |

## 输出

| 产出物 | 路径 | 必须 |
|---|---|---|
| 更新后的 L1 报告（含复杂度评级） | `docs/codearch/modules/<module_name>.md` | ✅ |
| 依赖关系图 | 总体报告中的 Mermaid 图 | ✅ |

## 核心任务

### 任务 1: 客观证据采集 [可并行，每模块独立]

对每个模块，按 [objective_metrics](../definitions/objective_metrics.md) 定义采集模块级客观指标：

```bash
MODULE_PATH="src/<module>"

# 代码行数
find $MODULE_PATH -name "*.cpp" -o -name "*.h" -o -name "*.c" | xargs wc -l | tail -1

# 类/结构体数量
grep -rn "^class \|^struct " $MODULE_PATH --include="*.h" | wc -l

# 公开头文件数量
ls $MODULE_PATH/*.h 2>/dev/null | wc -l

# 外部依赖类型扫描 — 线程/并发
grep -rn "pthread\|std::thread\|mutex\|atomic" $MODULE_PATH --include="*.cpp" --include="*.h" | head -3

# 外部依赖类型扫描 — I/O
grep -rn "fopen\|ifstream\|socket\|connect" $MODULE_PATH --include="*.cpp" --include="*.c" | head -3

# 外部依赖类型扫描 — 硬件/SDK
grep -rn "ioctl\|cuda\|opencl\|tensorrt" $MODULE_PATH --include="*.cpp" --include="*.h" | head -3

# 锁/互斥量数量（用于密度计算）
grep -rn "std::mutex\|pthread_mutex\|std::atomic\|std::lock_guard\|std::unique_lock" $MODULE_PATH --include="*.h" --include="*.cpp" | wc -l
```

将所有采集结果记录到 L1 报告中。

**密度信号计算**（基于上述采集结果）：

从 `docs/codearch/engineering_metadata.md`「复杂度阈值配置」读取工程总有效代码行数 T，然后计算：

- **并发密度** = 锁/互斥量数量 ÷ (模块代码行数 / 1000)，> 3 个/千行为高信号
- **接口密度** = 类/结构体数量 ÷ (模块代码行数 / 1000)，> 5 个/千行为高信号
- **外部依赖广度** = 命中的外部依赖类型种类数（I/O、网络、线程/并发、硬件/SDK、全局状态），≥ 3 种为高信号
- **模块体量占比** = 模块代码行数 ÷ T，> 30% 为高信号

将密度信号计算结果一并记录到 L1 报告中，供维度 f 使用。

### 任务 2: 结构化主观推理 [可并行，每模块独立]

对每个模块，按 [complexity_levels](../definitions/complexity_levels.md) 进行 5 个维度的结构化推理：

#### 维度 a) 代码规模复杂度

- **引用证据**: 代码行数、类/结构体数量（来自任务 1）
- **主观评估**: 代码是高密度（每段承载独立语义）还是低密度（模板化/重复代码）？
- **维度结论**: 低 / 中 / 高 / 极高

#### 维度 b) 依赖交互复杂度

- **引用证据**: 依赖数量、外部依赖类型（来自任务 1）
- **主观评估**: 交互模式是简单调用还是复杂回调/事件/状态同步？
- **维度结论**: 低 / 中 / 高 / 极高

#### 维度 c) 并发复杂度

- **引用证据**: mutex 数量、线程使用情况（来自任务 1）
- **主观评估**: 锁的保护范围是否明确且局部？是否存在跨锁协调？是否存在隐式时序约束？
- **维度结论**: 低 / 中 / 高 / 极高

#### 维度 d) 认知复杂度

- 无需客观指标——纯模型判断
- 列出向新开发者解释该模块所需的**核心概念**
- ≤ 3 个概念 → 低，4–6 → 中，7–10 → 高，> 10 → 极高
- **维度结论**: 低 / 中 / 高 / 极高

#### 维度 e) 可拆分性评估

- 是否存在清晰的拆分边界？是否存在多个独立职责？
- 是否存在独立的数据流？独立的并发区域？
- **维度结论**: 可拆 / 不可拆 / 需评估

#### 维度 f) 复杂度密度

- **引用证据**: 任务 1 中计算的 4 个密度信号值
- **主观评估**: 统计触发高信号的数量，判断该模块的复杂度是否集中
- 此维度用于捕获"代码量不大但复杂度高度集中"的模块
- **维度结论**: 低（0 高信号）/ 中（1 高信号）/ 高（2 高信号）/ 极高（≥3 高信号）

### 任务 3: 客观指标地板检查

从 `docs/codearch/engineering_metadata.md`「复杂度阈值配置」读取阈值参数，按 [objective_metrics](../definitions/objective_metrics.md) 中的强制评估触发阈值执行地板检查：

| 触发条件 | 要求 |
|---|---|
| 代码 > **地板阈值_高** 且评级 < 高 | 必须给出充分理由，否则提升至「高」 |
| 维度 f 密度信号 ≥ 3 个且评级 < 高 | 密度地板：复杂度高度集中，必须提升至「高」 |
| 硬件/SDK 依赖且评级 < 极高 | 必须给出充分理由，否则提升至「极高」 |
| 代码 > **拆分评估阈值** 或类 > 10 个 | 必须填写拆分评估（即使评级为「中」） |

### 任务 4: 最终评级与拆分说明

1. 综合维度 a–f 的结论 + 地板检查 → 确定最终评级
2. 撰写一段话的评级理由
3. 若最终评级与旧版机械规则可能给出的结论不同，**必须**说明差异原因
4. 对**极高**模块：必须填写拆分说明（**待拆** / 不拆）
5. 对**高**模块且满足触发条件：必须填写拆分说明（**待拆** / 不拆）
6. 将完整的结构化推理过程更新到 L1 报告

> **P2 禁写「已拆」**：P2 阶段的拆分说明**仅允许写「待拆」或「不拆」**，**禁止写「已拆」**。原因：P2 不创建子模块报告文件，无权声称拆分已完成。若判定模块应当拆分，必须写「待拆（拆分方案：...）」，由 P3 触发 P1 重入来执行实际拆分。详见 [complexity_levels — 拆分结论格式](../definitions/complexity_levels.md)。

### 任务 5: 依赖关系整体分析 [串行]

所有模块评级完成后，执行整体依赖分析：

1. 分析跨模块 `#include` 和链接依赖
2. 识别依赖类型（调用 / 数据传递 / 事件通知 / 配置读取 / 继承）
3. 识别 **HUB 模块**（被 ≥ 3 个其他模块依赖的模块）
4. 识别所有权转移边（ownership transfer edges）
5. 在总体报告中生成/更新 Mermaid 依赖关系图
6. 将精化后的依赖信息回填到各 L1 报告的依赖概览

## 注意事项

| 类别 | 说明 |
|---|---|
| **DO** | 引用客观证据；使用结构化推理模板；执行地板检查；解释与机械规则的偏差 |
| **DON'T** | 不要跳过任何维度；不要仅凭机械规则决定评级；不要为了减少工作量而压低评级；**不要将拆分说明直接标为「已拆」——P2 阶段无权设置此状态**（详见 [complexity_levels](../definitions/complexity_levels.md)） |

---

## 反偷懒提示

> 你的评级将在 P3 收敛评估中被**自动化脚本**交叉验证。P3 会程序化检查每个模块的代码行数与评级是否符合地板规则（代码行数超过**地板阈值_高**的模块必须至少评「高」，密度信号 ≥3 的模块同理）。阈值从 `engineering_metadata.md` 动态读取，不是固定值。若脚本检测到违规，P3 将**自动判定未收敛**并要求返回 P2 重新评级——你无法通过主观推理绕过地板规则。请基于证据诚实评级。
