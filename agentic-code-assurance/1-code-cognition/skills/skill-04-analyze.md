# Skill 04: 深度分析（L1 → L2 升级）

> **触发**: Phase 04 指示  
> **目标**: 为所有模块执行完整深度分析，将 L1 报告升级为 L2 报告  
> **预计耗时**: 1–8 小时（视工程规模而定）

---

## 输入

- 所有模块的 L1 报告
- 工程源代码
- `docs/codearch/engineering_metadata.md`

## 输出

| 产出物 | 路径 | 必须 |
|---|---|---|
| L2 报告（所有模块） | `docs/codearch/modules/<module_name>.md` | ✅ |

参考：[module-report-L2 模板](../templates/module-report-L2.md)、[产出路径与报告结构约定](../definitions/output_structure.md)

---

## 核心任务

**执行顺序**: 任务 1–6、8–9 可按模块并行执行；任务 7 需在所有模块完成后串行执行；任务 10 在全部完成后执行。

### 任务 1: 关键设计要点 [每模块，可并行]

通过阅读模块关键源文件，识别 3–5 个关键设计要点：

- 主要入口函数/类
- 核心数据结构
- 线程/并发假设
- 错误处理策略
- 扩展点

### 任务 2: 代码特征扫描 [每模块，可并行]

按 [output_structure](../definitions/output_structure.md) 的要求扫描并记录：

- **内存管理**: RAII / 智能指针 / 手动管理 / 混合
- **并发模型**: 是否多线程、同步机制
- **I/O 操作**: 文件 I/O、网络 I/O
- **外部数据处理**: 是否处理外部输入、数据来源
- **错误处理**: 策略（返回错误码 / 异常 / 混合）

```bash
MODULE_PATH="src/<module>"
# 内存管理
grep -rn "malloc\|calloc\|new \|delete \|unique_ptr\|shared_ptr" $MODULE_PATH --include="*.cpp" | head -5
# 并发
grep -rn "pthread\|std::thread\|mutex\|atomic" $MODULE_PATH --include="*.cpp" --include="*.h" | head -5
# I/O
grep -rn "fopen\|ifstream\|socket\|connect" $MODULE_PATH --include="*.cpp" --include="*.c" | head -5
# 错误处理
grep -rn "throw\|catch\|errno\|perror" $MODULE_PATH --include="*.cpp" | head -5
```

### 任务 3: 关键代码位置索引 [每模块，可并行]

为每个模块建立关键代码位置索引：

- **主要入口点**：函数/类、file:line、简要说明
- **外部数据入口**（若有）：含首道校验位置
- **错误/异常路径**（关键）：触发条件、清理逻辑位置、是否完整

### 任务 4: 关键数据流路径分析 [每模块，可并行]

- 识别模块的关键数据入口点
- 追踪数据从入口到出口的流转路径
- 标记所有权转移点（`std::move`、裸指针交接等）
- 填写数据流表格：路径名、起点、经过、终点、涉及的所有权转移

### 任务 5: 生命周期与所有权模型 [每模块，可并行]

仅针对涉及手动内存管理或跨函数/跨模块生命周期的模块：

- 追踪关键对象/资源：创建位置、销毁位置、所有权模型、转移点
- 无动态分配的模块：标注「无需记录」

### 任务 6: 并发不变量提取 [每多线程模块，可并行]

仅针对多线程模块：

- 受保护数据/结构
- 保护机制（mutex / atomic / 读写锁等）
- 锁获取顺序约定
- 线程上下文假设

单线程模块：填写「本模块为单线程模型，无并发不变量」

### 任务 7: 模块关系结构化 [串行，需所有模块完成]

为**所有**模块填写「与其它模块的关系」结构化表格：

- **上游依赖**（本模块依赖的模块）：模块名、依赖类型、是否涉及跨模块所有权转移、简要说明
- **下游依赖**（依赖本模块的模块）：同上格式

额外工作：

- 识别枢纽模块（被 ≥3 个模块依赖）
- 更新总体报告的模块依赖图，补充精化后的依赖信息

### 任务 8: 信息来源与使用示例 [每模块，可并行]

- 记录信息来源（README、测试文件、头文件注释、代码分析等）
- 提供至少一个使用示例（从测试/示例代码提取或 Agent 编写，须标注来源）
- 复杂度为 高/极高 的模块：填写「验证状态」章节

### 任务 9: 父模块子模块索引 [仅已拆分的父模块]

对已拆分的父模块，追加「子模块索引」章节：

- 列出子模块：名称、路径/范围、复杂度、报告链接
- 注明拆分依据（按功能职责 / 按数据流阶段 / 按并发边界）

### 任务 10: 升级标记

- 将报告头部从 `报告层级: L1（迭代阶段）` 修改为 `报告层级: L2（完整版）`
- 验证所有 L2 必填章节均已填写
- 更新总体报告的「技术特征统计」表格（基于所有模块的代码特征汇总）

## 验收标准

- [ ] 所有模块报告均为 L2 层级
- [ ] 每份 L2 报告包含全部必填章节（参见 [module-report-L2 模板](../templates/module-report-L2.md)）
- [ ] 所有「与其它模块的关系」表格已填写
- [ ] 总体报告的依赖图与技术特征统计已更新

### 验收检查

```bash
# 确认所有报告已升级为 L2
for f in docs/codearch/modules/*.md; do
  grep -q "L2" "$f" && echo "OK $(basename $f)" || echo "STILL L1 $(basename $f)"
done
```

## 注意事项

| 类别 | 说明 |
|---|---|
| **DO** | 以代码证据为依据，标注 file:line 位置 |
| **DO** | 优先从现有文档/测试中提取信息 |
| **DO** | 代码特征只记录客观事实，不做风险判断 |
| **DON'T** | 不要猜测模块职责或依赖关系——须有代码证据 |
| **DON'T** | 不要跳过高复杂度模块的验证状态填写 |
| **DON'T** | 不要在代码特征中写风险判断（仅记录事实） |
