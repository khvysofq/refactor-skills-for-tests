# Skill 02-DD: 模块深入分析（递归拆分）

> **触发条件**：由 [Skill 02 Task 5（分解审视）](skill-02-modules.md) 触发——审视发现某模块复杂度为「高」或「极高」且满足 [拆分建议触发条件](../definitions/complexity_levels.md) 中的拆分标准；或由本 Skill 上一轮产生的子模块报告完成后，若子模块复杂度仍为「高/极高」且当前层级未达递归上限。
>
> **目标**：对单个高/极高复杂度模块进行内部子结构识别，生成粒度更细的子模块独立报告，原有父模块报告全部保留并追加子模块索引章节。

---

## 输入

- **父模块报告**：`docs/codearch/modules/<parent>.md`（必须已存在）
- **父模块源码目录**：报告中「模块名与路径」所指向的代码路径
- **当前递归层级**：由调用方传入（顶层模块 = 第 1 层，其子模块 = 第 2 层，以此类推）

---

## 输出

| 产出 | 路径 | 说明 |
|------|------|------|
| 子模块报告（每个子模块一份） | `docs/codearch/modules/<parent>_<submodule>.md` | 与顶层模块报告结构完全相同，独立存在 |
| 父模块报告更新 | `docs/codearch/modules/<parent>.md` | 追加「子模块索引」章节，原有内容全部保留 |
| 总体报告模块列表更新 | `docs/codearch/overall_report.md` | 子模块以缩进方式插入父模块条目下方 |

---

## 递归控制规则

在执行任何步骤前，先确认以下约束：

- **递归上限**：默认最大 **3 层**（顶层模块 = 第 1 层 → 一级子模块 = 第 2 层 → 二级子模块 = 第 3 层）。若当前层级已为第 3 层，跳过步骤 2–4，直接在报告中注明「已达递归上限，当前粒度为最细层级」，然后执行步骤 5（更新父模块报告）后返回。
- **收敛条件**：某子模块的复杂度评级 ≤ 中，则该子模块不再触发新一轮深入分析（即使层级未达上限）。
- **平铺命名**：多级递归时文件名继续平铺拼接，例如第 2 层子模块 `communicator_session` 的子模块命名为 `communicator_session_handler.md`，不使用目录嵌套。

---

## 执行步骤

### 步骤 1：加载父模块报告，理解当前认知

1. 读取 `docs/codearch/modules/<parent>.md`，重点关注以下章节作为拆分依据：
   - **职责描述**：有几个明显不同的功能职责？
   - **关键数据流路径**：有几条互不相交的独立数据流？
   - **并发不变量**（若有）：有几个独立的锁保护区域？
   - **代码特征**：内存管理方式、并发模型，为后续子模块评级提供参考

2. 根据报告中的「复杂度评级 - 拆分说明」，了解之前对拆分方向的初步判断（若已有）。

3. 确认父模块对应的源码路径（从报告头部「路径」字段），用于后续代码探索。

---

### 步骤 2：识别内部子结构（拆分边界分析）

按以下优先级顺序尝试识别拆分边界，选取最清晰的一种作为本次拆分依据：

#### 方法一：按功能职责拆分（优先级最高）

在父模块的源码目录中，按类/文件的职责归类，识别 2–4 个功能内聚的子集。

```bash
MODULE_PATH="<父模块路径>"

# 列出模块内所有头文件，按命名前缀分组，观察命名规律
ls $MODULE_PATH/*.h 2>/dev/null | sort

# 查看类定义，理解类之间的职责分工
grep -rn "^class " $MODULE_PATH/ --include="*.h" | sort

# 统计各文件行数，识别主要组成部分
wc -l $MODULE_PATH/*.cpp $MODULE_PATH/*.h 2>/dev/null | sort -rn | head -20
```

判断标准：若能识别出 2–4 个职责彼此独立（修改其中一个不影响其他）的子集，则按职责拆分。

#### 方法二：按数据流阶段拆分

从父模块报告的「关键数据流路径」章节，识别数据处理的阶段边界：
- **阶段一**：外部数据接收 / 输入解析
- **阶段二**：核心处理逻辑
- **阶段三**：结果输出 / 事件派发

```bash
# 追踪数据入口函数，理解其调用链阶段
grep -rn "recv\|read\|parse\|deserialize" $MODULE_PATH/ --include="*.cpp" | head -10
grep -rn "send\|write\|dispatch\|emit\|notify" $MODULE_PATH/ --include="*.cpp" | head -10
```

判断标准：若能识别出 2–3 个数据处理阶段，且各阶段代码文件边界清晰，则按数据流阶段拆分。

#### 方法三：按并发边界拆分

从父模块报告的「并发不变量」章节，识别独立的锁保护区域：

```bash
# 查找锁变量定义，理解锁的粒度与保护范围
grep -rn "mutex\|pthread_mutex_t\|std::mutex" $MODULE_PATH/ --include="*.h" --include="*.cpp" | head -10

# 查找线程入口函数，理解各线程的职责
grep -rn "pthread_create\|std::thread\|async" $MODULE_PATH/ --include="*.cpp" | head -10
```

判断标准：若存在 2–3 个相互独立的并发保护区域（各自使用独立的锁），则按并发边界拆分。

> **注意**：若上述三种方法均无法识别清晰的拆分边界（模块内部高度耦合），在「复杂度评级 - 拆分说明」中注明「内部耦合度过高，无法识别清晰边界，维持当前粒度」，并跳过步骤 3–4，直接返回。

---

### 步骤 3：确定子模块划分方案，为每个子模块命名

1. 基于步骤 2 的分析，确定 2–4 个子模块的划分方案。

2. **命名规则**：`<parent>_<描述性名称>`
   - 用下划线连接，全部小写，不含空格
   - 描述性名称应反映子模块的核心职责（如 `poller`、`session`、`conn`、`parser`、`dispatcher`）
   - 多级递归时继续平铺：`<grandparent>_<parent>_<child>`（如 `communicator_session_handler`）

3. 记录每个子模块的划分信息：
   - 子模块名称（完整名，如 `communicator_poller`）
   - 对应的源码文件列表（具体文件名或 glob，如 `src/kernel/Poller.h`、`src/kernel/Poller.cc`）
   - 主要职责一句话描述

4. 评估子模块间的依赖关系（哪些子模块会调用哪些），为步骤 4 的报告撰写做准备。

---

### 步骤 4：为每个子模块生成独立报告

对每个子模块，按 [模块报告模板](../templates/module-report.md) 生成完整的独立报告，保存到 `docs/codearch/modules/<parent>_<submodule>.md`。

报告内容要求：

1. **报告头部**：模块名填写子模块完整名（如 `communicator_poller`），路径填写对应的源码文件范围

2. **职责描述**：聚焦该子模块自身的核心职责，不描述整个父模块的职责

3. **边界**：记录该子模块的输入/输出（包括来自兄弟子模块的数据传递）

4. **依赖**：
   - 内部依赖须同时记录：**同级兄弟子模块**的依赖关系 + 继承自父模块的**其他顶层模块**依赖
   - 外部依赖：记录该子模块直接使用的外部系统调用/库

5. **「与其它模块的关系」结构化表格**：
   - 上游依赖表：同时包含兄弟子模块和外部顶层模块
   - 下游依赖表：同时包含兄弟子模块和外部顶层模块

6. **复杂度评级**：独立评级（不继承父模块的评级）；若仍为「高/极高」且满足拆分触发条件，填写拆分说明（本 Skill 会在步骤 4 完成后检查是否触发下一层递归）

7. 其余章节（代码特征、关键代码位置索引、关键数据流路径、生命周期与所有权模型、并发不变量、使用示例、信息来源）均按模板正常填写，聚焦该子模块自身的代码范围

**步骤 4 完成后：检查是否触发递归**

- 对每个新生成的子模块报告，检查其复杂度评级
- 若复杂度为「高/极高」**且**满足拆分触发条件**且**当前层级 < 递归上限（3 层）：
  - 以该子模块为新的「父模块」，重新从步骤 1 开始执行
- 若复杂度 ≤ 中，或不满足拆分触发条件，或已达递归上限：
  - 继续执行步骤 5

---

### 步骤 5：更新父模块报告和总体报告

#### 5a. 更新父模块报告（追加章节，原有内容全部保留）

在 `docs/codearch/modules/<parent>.md` 末尾追加以下章节（不修改任何已有内容）：

```markdown
---

## 子模块索引（深入分析结果）

> 本模块已于 [分析日期/轮次] 进行深入分析，识别出以下子模块。
> 原有报告内容作为顶层概览保留，子模块报告提供更细粒度的分析。
> 拆分依据：[按功能职责 / 按数据流阶段 / 按并发边界]

| 子模块名称 | 路径/范围 | 复杂度 | 报告链接 |
|-----------|---------|--------|---------|
| <parent>_<sub1> | <源码文件列表> | 低/中/高/极高 | [报告](<parent>_<sub1>.md) |
| <parent>_<sub2> | <源码文件列表> | 低/中/高/极高 | [报告](<parent>_<sub2>.md) |
```

若某子模块进一步被拆分（多级递归），在对应行后注明：

```markdown
| <parent>_<sub2> | ... | 高（已进一步深入分析） | [报告](<parent>_<sub2>.md) |
| ↳ <parent>_<sub2>_<subsub1> | ... | 中 | [报告](<parent>_<sub2>_<subsub1>.md) |
```

#### 5b. 更新总体报告「模块列表」

在 `docs/codearch/overall_report.md` 的模块列表中，找到父模块所在行，在其下方以缩进方式插入子模块条目：

```markdown
| <parent> | src/path/... | 极高（已深入分析） | [链接](modules/<parent>.md) |
| ↳ <parent>_<sub1> | src/path/sub1... | 中 | [链接](modules/<parent>_<sub1>.md) |
| ↳ <parent>_<sub2> | src/path/sub2... | 高 | [链接](modules/<parent>_<sub2>.md) |
```

同时将父模块的复杂度列更新为「原等级（已深入分析）」，例如「极高（已深入分析）」。

#### 5c. 更新总体报告「模块依赖图」（Mermaid 图）

- 在 Mermaid 图中，将父模块节点标注为 `[已拆分]`
- 为每个子模块添加节点，并绘制兄弟子模块之间的依赖边（根据步骤 4 中子模块报告里的依赖关系填写）

---

## 验收标准

深入分析完成后，验证以下条件全部满足：

- [ ] 每个已识别的子模块均有对应的 `docs/codearch/modules/<parent>_<submodule>.md` 报告文件
- [ ] 每份子模块报告含完整的必须章节（与顶层模块报告要求相同）
- [ ] 每份子模块报告含「与其它模块的关系」结构化表格，包含兄弟子模块间的依赖
- [ ] 父模块报告末尾已追加「子模块索引」章节，原有内容完整保留
- [ ] 总体报告模块列表中父模块条目下方已插入子模块条目（带 `↳` 缩进）
- [ ] 所有叶子子模块的复杂度 ≤ 中，或已注明「已达递归上限」或「内部耦合无法拆分」

```bash
# 检查子模块报告是否存在（以 communicator 为例）
ls docs/codearch/modules/communicator_*.md 2>/dev/null

# 检查父模块报告是否含子模块索引章节
grep -q "子模块索引" docs/codearch/modules/<parent>.md && echo "PASS" || echo "FAIL"

# 检查总体报告是否含子模块条目
grep -q "↳" docs/codearch/overall_report.md && echo "PASS (含缩进子模块)" || echo "CHECK"
```

## 状态跳转

| 结果 | 下一步 |
|------|--------|
| 全部子模块复杂度 ≤ 中，或已达递归上限/无法拆分 | 返回 [Skill 02 Task 5](skill-02-modules.md)，以当前所有模块（含新生成的子模块报告）为输入重新执行分解审视 |
| 某子模块仍为高/极高 且层级 < 3 且满足拆分触发条件 | 以该子模块为新的父模块，从步骤 1 重新开始执行本 Skill |

---

**完成后**：返回 [Skill 02 Task 5](skill-02-modules.md) 执行最终分解审视
