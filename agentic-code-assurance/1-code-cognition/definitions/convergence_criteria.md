# 收敛条件与迭代控制

> **阅读时机**：P3 收敛评估时阅读。

---

## 目的

每完成一轮 P1→P2 迭代后，评估当前模块分解是否已收敛。若已收敛，进入 P4 深入分析；若未收敛，产出**再入列表**——列出需要重新进入 P1 进行进一步分解的模块。

---

## 收敛条件

一个**叶子模块**满足以下任一条件即视为已收敛：

| 条件 | 说明 |
| ---- | ---- |
| **(a) 复杂度 ≤ 中，且未命中地板规则** | 评级为中或低，**且**代码行数 ≤ **地板阈值_高**（见 `engineering_metadata.md`「复杂度阈值配置」）**且**维度 F 密度高信号数 < 3（或已提供充分的地板豁免理由）。若评级为「中」但代码行数超过地板阈值_高或密度高信号 ≥ 3 且无豁免，则判定为「评级疑似偏低」，加入再入列表要求重新 P2 评级 |
| **(b) 复杂度为高/极高，拆分状态为「已拆」且子模块报告文件已验证存在** | 「已拆」不仅要求文本声明，还需通过自动化脚本验证子模块 `.md` 文件实际存在于 `docs/codearch/modules/`（子模块成为新的叶子节点继续评估） |
| **(c) 复杂度为高/极高，持有合格的「不拆」理由** | 理由必须包含具体技术论证（跨模块依赖、双向耦合、共享关键状态），详见下方[「不拆」理由格式要求](#不拆理由格式要求)。代码量超过**逐目录论证阈值**（见 `engineering_metadata.md`）的模块须逐子目录论证，详见 [complexity_levels.md](complexity_levels.md) |
| **(d) 递归深度已达上限** | 默认上限 3 层：顶层=1，首次拆分=2，再次拆分=3 |

**整体收敛判定**：当所有叶子模块均满足上述条件之一，**且**自动化验证门禁（地板规则合规性 + 已拆报告存在性）全部通过时，本轮迭代收敛。

---

## 迭代控制规则

| 规则 | 说明 |
| ---- | ---- |
| **最大迭代轮数** | 3 轮（从首次执行 P1 开始计数） |
| **达到上限的处理** | 将剩余高复杂度模块标注为「已达迭代上限」，直接进入 P4 |
| **进度要求** | 每轮迭代必须产生进展——至少一个模块完成拆分或获得合格的「不拆」理由 |
| **无进展强制收敛** | 若某轮迭代未产生任何进展，强制判定收敛，进入 P4 |

---

## P3 评估维度

对当前模块划分逐项检查。**维度 0 为自动化硬门禁，必须首先执行**：

| 维度 | 检查要点 | 检查方式 |
| ---- | -------- | -------- |
| **0a. 地板规则合规性**（硬门禁） | 所有模块的最终评级符合客观指标地板规则：代码行数超过**地板阈值_高**（见 `engineering_metadata.md`）的模块评级不低于「高」，维度 F 密度高信号 ≥ 3 的模块评级不低于「高」，涉及硬件/SDK 的模块评级不低于「极高」 | **自动化脚本**（见 [Skill 03](../skills/skill-03-converge.md) Task 0） |
| **0b. 已拆报告存在性**（硬门禁） | 所有拆分状态为「已拆」的模块，其声称的子模块报告文件均实际存在于 `docs/codearch/modules/` 目录中 | **自动化脚本**（见 [Skill 03](../skills/skill-03-converge.md) Task 0） |
| **1. 叶子模块收敛** | 所有叶子模块均满足收敛条件 (a)/(b)/(c)/(d) | 逐模块检查 |
| **2. 拆分说明完整** | 所有触发拆分评估的高/极高模块均已标注终态拆分状态（「已拆」或「不拆」），不允许「待拆」或含糊表述残留 | 逐模块检查 |
| **3. 粒度一致** | 同层级模块之间不存在体量悬殊的情况 | 逐层级对比 |
| **4. 评级交叉验证** | 复核 P2 评级——结合维度 0a 的自动化结果，确认不存在客观指标应导致更高等级但实际评级偏低的模块（反偷懒检查） | 参考脚本输出 + 人工复核 |
| **5. 索引一致** | 总报告中的模块索引与磁盘上实际存在的模块报告一一对应 | 脚本验证 |

**结论**：全部通过 → **已收敛**；任一项不通过 → **未收敛**，须产出再入列表。**维度 0a/0b 不通过时无需继续后续维度检查**，直接判定未收敛。

## 未收敛时的处理

产出**再入列表**，明确每个未收敛模块的处置方式：

| 情况 | 处置 |
| ---- | ---- |
| 拆分说明为「待拆」 | 进入 P1，识别子模块并生成报告 |
| 复杂度评级疑似偏低（含地板规则违规） | 重新进入 P2，按客观指标重新评级 |
| 「已拆」但子模块报告文件缺失 | 进入 P1，生成缺失的子模块报告 |
| 模块报告缺失 | 进入 P1，生成对应模块报告 |

多个模块可在 P1 中并行处理。

## 不拆理由格式要求

当高/极高复杂度模块决定不拆分时，「不拆」理由**必须**包含以下至少一项具体技术论证：

1. **拆分将引入新的跨模块依赖**——说明拆分后哪些内部调用会变为跨模块调用，以及由此引入的依赖路径。
2. **内部存在双向耦合，无法划出干净边界**——指出具体的双向调用链或相互引用关系。
3. **共享关键内部状态，拆分后需暴露该状态**——说明哪些内部状态会被迫成为公开接口，以及带来的风险。

**以下理由不合格，不能作为「不拆」的充分依据**：

- ❌「各子模块职责清晰」
- ❌「保持完整性」
- ❌「通过接口解耦」
- ❌「已有子目录划分」
