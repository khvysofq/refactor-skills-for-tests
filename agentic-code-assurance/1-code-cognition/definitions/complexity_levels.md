# 模块复杂度等级定义

> **阅读时机**：进行模块复杂度评级时（执行 Skill 02 或 Phase 02 时）。

---

## 等级概览

| 等级 | 名称 | 典型特征                               |
| ---- | ---- | -------------------------------------- |
| 低   | 低   | 代码量小、依赖少、无外部 I/O/线程/硬件 |
| 中   | 中   | 中等规模或少量外部依赖                 |
| 高   | 高   | 大量依赖、多外部系统、并发/状态多      |
| 极高 | 极高 | 跨进程/硬件/强非确定性或历史债重       |

---

## 判定维度

对每个模块综合以下维度打分，再映射到上述等级：

### 1. 代码规模

| 规模 | 行数（参考） | 得分影响 |
| ---- | ------------ | -------- |
| 小   | < 500        | 降一档   |
| 中   | 500–3000     | 中性     |
| 大   | 3000–10000   | 升一档   |
| 超大 | > 10000      | 升两档   |

### 2. 内部依赖数量

- **少**：直接依赖 ≤ 3 个模块 → 中性或降一档
- **中**：4–8 个模块 → 中性
- **多**：> 8 个模块 → 升一档

### 3. 外部依赖类型

| 类型          | 示例                         | 影响         |
| ------------- | ---------------------------- | ------------ |
| 无/仅标准库   | 无文件/网络/时间/线程        | 中性或降一档 |
| I/O           | 文件、管道、标准流           | 升一档       |
| 时间/随机     | time, clock, rand            | 升一档       |
| 网络          | socket, HTTP, RPC            | 升一档       |
| 线程/并发     | pthread, std::thread, mutex  | 升一档       |
| 全局状态/单例 | static, extern, ::instance() | 升一档       |
| 硬件/SDK      | 厂商库、驱动接口             | 升两档       |

多种类型并存时可按叠加效果升档。

### 4. 控制流与结构复杂度

- 单一路径、无深层嵌套 → 中性
- 多分支、多错误路径、状态机 → 升一档
- 回调/异步/事件驱动、难以线性阅读 → 升一档

### 5. 历史债与可读性

- 注释少、命名混乱、魔法数多、强耦合 → 可额外升一档

---

## 等级映射规则

1. 从**中**起步，按上述维度调整。
2. 按以下规则确定最低等级（规则按优先级从高到低排列，取命中的最高等级）：
   - 2a. 任一维度达到「升两档」**且**同时存在 ≥1 个「升一档」维度 → 至少**极高**
   - 2b. 涉及硬件/跨进程通信/强非确定性（如随机+并发交叉） → 至少**极高**
   - 2c. 任一维度达到「升两档」（即使无其他升档维度） → 至少**高**
   - 2d. 无「升两档」但存在 ≥3 个独立的「升一档」维度 → 至少**高**
3. 所有维度均偏简单 → **低**。
4. 不确定时优先选**中**或**高**，并在模块报告中注明「评级依据简要说明」。

> **规则 2 说明**：规则 2a 确保「超大代码量 + 任一额外复杂维度」的模块不会被低评为「高」。例如代码量 > 10,000 行（升两档）且存在线程/并发（升一档），必须评为「极高」。

---

## 拆分建议触发条件

评级完成后，根据等级判断是否建议拆分。拆分建议须在模块报告的「复杂度评级」一节中明确说明（已拆 / 不拆的理由）。

### 强烈建议拆分（等级 = 极高）

满足以下**任一**条件时，强烈建议将模块拆分为 2–4 个子模块：

| 触发条件 | 说明 |
|----------|------|
| 代码量 > 5000 行 | 单模块过大，静态分析成本高，下游路径追踪困难 |
| 内部存在 ≥2 个明显不同的功能职责 | 违反单一职责，难以隔离测试与审查 |
| 内部存在 ≥2 条完全独立的关键数据流路径 | 数据流不相交，说明模块可沿数据流边界拆分 |

**拆分目标**：每个子模块复杂度评级目标 ≤ 高，且子模块间依赖方向清晰（尽量单向）。

**拆分维度参考**（按优先级）：
1. 按**功能职责**：将负责不同业务功能的代码分入独立子模块
2. 按**数据流阶段**：将输入解析、核心处理、输出/事件派发分为不同子模块
3. 按**并发边界**：将需要加锁保护的共享状态与无锁的纯计算逻辑分离

### 须评估拆分（等级 = 高）

满足以下**全部**条件时，**须**评估是否可拆分，并在模块报告中填写拆分说明（已拆/不拆/待拆）：

| 触发条件 |
|----------|
| 代码量 > 3000 行 |
| 内部职责数 > 1（即模块承担了多个可独立描述的功能） |

若评估后决定**不拆分**，须在模块报告「复杂度评级」一节中写明**合格的不拆理由**（见下方「在模块报告中的填写」一节的理由格式要求）。

### 不建议拆分（等级 = 中或低）

无论代码量多少，中/低复杂度模块通常不需要拆分，维持原有粒度即可。

## 在模块报告中的填写

在 [模块报告](../templates/module-report.md) 的「复杂度评级」一节中：

- 填写最终等级：**低 / 中 / 高 / 极高**
- 可选：用一句话说明主要依据（如「依赖 6 个内部模块 + 文件与网络 I/O」）
- **评级维度记录**（等级为高或极高时**必填**）：必须逐维度列出得分影响（升一档/升两档/中性/降一档），格式为简表或一行文字。示例：`代码规模: 升两档（约20,000行）; 线程/并发: 升一档; I/O: 升一档; 控制流: 升一档 → 至少极高`。此记录用于分解审视时的评级复核，确保等级与维度得分一致。
- **拆分说明**（等级为极高时必填，等级为高且满足拆分触发条件时**必填**）：结论 **必须** 为以下三种之一，不允许使用「建议拆分」「建议进一步分析」等模糊表述作为终态：

| 结论 | 格式要求 | 后续动作 |
|------|---------|---------|
| **已拆** | `已拆（子模块：xxx_a, xxx_b, ...）` — 须列出子模块名称，且对应的子模块报告已生成 | 无（拆分已完成） |
| **不拆** | `不拆（理由：...）` — 须给出**合格的**具体技术理由（见下方理由格式要求） | 无（明确决策不拆） |
| **待拆** | `待拆（拆分方案：xxx_a, xxx_b, ...）` — 已识别拆分方案但尚未执行 | **此状态下分解审视不可标记为「通过」**，须执行 [Skill 02-DD](../skills/skill-02-drilldown.md) 完成深入分析后将结论更新为「已拆」 |

> **重要**：拆分说明为「待拆」的模块会阻断分解审视通过（见 [分解审视约定](decomposition_review.md)）和 Phase 02 验收通过（见 [Phase 02](../phases/02-modules.md)）。Agent 必须执行 Skill 02-DD 将其推进为「已拆」或重新评估为「不拆」后，流程才能继续。

### 「不拆」理由格式要求

「不拆」理由**必须**包含以下至少一项具体技术论证：

- **(a) 拆分引入新依赖**：说明拆分后会新增哪些跨模块依赖或共享状态访问路径变化（如「ScenarioManager 与 ExecsManager 共享 running_topology_ 状态，拆分后需新增跨模块接口暴露该状态」）
- **(b) 内部强双向耦合**：指出模块内部哪些类/函数之间存在强双向调用关系，使得无法在不增加跨模块依赖的前提下划分边界（如「Parser 与 Validator 相互回调，拆分后两者需循环依赖」）
- **(c) 共享关键内部状态**：指出模块内多个子目录/子模块共享哪些关键内部数据结构或状态，拆分会导致该状态需要跨模块暴露（如「三个子模块共享内部缓存池 BufferPool，拆分后需将其提升为独立模块或公开内部接口」）

**不合格理由示例**（以下表述不被接受，分解审视将判定为不通过）：

| 不合格表述 | 原因 |
|-----------|------|
| "各子模块职责清晰" | 职责清晰恰恰说明拆分边界明确，是拆分可行的论据，不构成不拆理由 |
| "作为 XX 层需要保持完整性" | 过于笼统，未说明拆分的具体负面后果 |
| "通过接口解耦" | 接口解耦说明拆分成本低，不构成不拆理由 |
| "已有子目录划分" | 目录划分是代码组织方式，不等同于模块报告层面的拆分 |
