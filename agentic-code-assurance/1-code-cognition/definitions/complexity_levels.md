# 模块复杂度等级定义（证据驱动）

> **阅读时机**：进行模块复杂度评级时（执行 Skill 02 或 Phase 02 时）。

---

## 等级概览

| 等级 | 名称 | 典型特征 |
| ---- | ---- | -------- |
| 低   | 低   | 代码量小、逻辑单一、依赖少、无外部 I/O/线程/硬件，一屏可概览核心逻辑 |
| 中   | 中   | 中等规模或少量外部依赖，核心概念 4–6 个，单一职责但有一定交互复杂度 |
| 高   | 高   | 大量依赖、多外部系统、并发/状态多，需理解多条交互路径才能把握全貌 |
| 极高 | 极高 | 跨进程/硬件/强非确定性、多维度同时复杂、或历史债导致认知成本极高 |

---

## 证据采集

对每个模块，**必须**先采集以下客观指标，作为后续结构化推理的输入。所有指标均须在模块报告中记录实际数值。

### 1. 代码行数

统计模块范围内的有效代码行数（不含空行和注释行）。

```bash
# 方式一：cloc（推荐，自动排除空行和注释）
cloc src/module/ --by-file --quiet

# 方式二：wc -l（粗略统计，含空行和注释）
find src/module/ -name "*.cpp" -o -name "*.h" | xargs wc -l
```

### 2. 类/结构体数量

```bash
grep -rE "^(class|struct)\s+\w+" src/module/ --include="*.h" --include="*.hpp" | wc -l
```

### 3. 公开头文件数量

```bash
find src/module/include/ -name "*.h" -o -name "*.hpp" | wc -l
# 若无 include 子目录，统计模块顶层头文件
find src/module/ -maxdepth 1 -name "*.h" -o -name "*.hpp" | wc -l
```

### 4. 内部依赖数量

统计本模块直接依赖的其他内部模块数（通过 `#include` 路径或构建系统依赖图获取）。

```bash
# 从 include 路径推断（需根据项目结构调整路径模式）
grep -rhE '#include\s+"' src/module/ --include="*.cpp" --include="*.h" \
  | sed 's|.*#include\s*"||; s|".*||; s|/.*||' \
  | sort -u
```

### 5. 外部依赖类型

逐项检查以下外部依赖类型是否存在，并记录具体使用位置：

| 依赖类型 | 检查方式示例 |
| -------- | ------------ |
| 文件 I/O | `grep -rn "fopen\|ifstream\|ofstream\|open(" src/module/` |
| 网络 I/O | `grep -rn "socket\|connect\|accept\|send\|recv\|http\|grpc" src/module/` |
| 线程/并发 | `grep -rn "thread\|pthread\|async\|future" src/module/` |
| 时间/随机 | `grep -rn "time\|clock\|rand\|random\|chrono" src/module/` |
| 硬件/SDK | `grep -rn "ioctl\|mmap\|DMA\|register\|vendor_sdk" src/module/` |
| 全局状态 | `grep -rn "static\s.*=\|extern\|::instance\|singleton" src/module/` |

### 6. 锁/互斥量数量（并发模块）

```bash
grep -rcE "mutex|lock_guard|unique_lock|shared_lock|atomic|semaphore|condition_variable" \
  src/module/ --include="*.cpp" --include="*.h"
```

## 结构化主观推理

采集客观指标后，对以下 **5 个维度**逐一进行结构化推理。每个维度的评估**必须**包含三部分：

1. **引用的客观证据**：列出本维度依据的具体指标数值或代码事实
2. **主观评估**：基于证据的分析判断（不是简单的数值映射）
3. **维度结论**：该维度的等级判定（低 / 中 / 高 / 极高）

### 维度 A：代码规模复杂度

**核心问题**：模块的信息密度如何？不是"有多少行代码"，而是"每段代码是否承载独立语义"。

评估要点：
- 代码是否大量重复/模板化/自动生成？若是，实际语义复杂度远低于行数暗示的复杂度
- 同等行数下，每个函数/类是否承担独立职责？还是存在大量结构相似的处理逻辑？
- 阅读代码时是否需要逐行理解，还是可以"见一知十"？

示例推理：
> **引用证据**：cloc 统计 12,000 行，其中 8,000 行为 40 个协议解析函数，结构高度相似（均为 header→payload→checksum 三段式）。
> **主观评估**：虽然行数超过万行，但 80% 代码为重复模式，理解一个即可推断其余。真正需要独立理解的核心逻辑约 4,000 行。
> **维度结论**：中

### 维度 B：依赖交互复杂度

**核心问题**：模块与其依赖之间的交互模式有多复杂？不是"依赖了几个模块"，而是"交互是否线性可预测"。

评估要点：
- 依赖交互是简单的函数调用（请求-响应），还是涉及回调、事件驱动、状态同步？
- 是否存在隐式依赖（如通过全局状态、事件总线、共享内存间接交互）？
- 依赖方向是否清晰单向？是否存在循环依赖或双向回调？

示例推理：
> **引用证据**：内部依赖 7 个模块；其中 5 个为单向调用（调用其 API 获取结果），2 个涉及回调注册（模块 X 注册回调到 EventBus，模块 Y 通过共享队列异步传递数据）。
> **主观评估**：虽然依赖数量为 7，但 5 个为简单调用，真正增加复杂度的是 2 个异步交互路径，需要理解事件顺序和队列语义。
> **维度结论**：中

### 维度 C：并发复杂度

**核心问题**：并发控制的语义复杂度如何？不是"有几把锁"，而是"锁的保护范围和协作模式是否清晰"。

评估要点：
- 每把锁的保护范围是否明确且局部？还是多把锁需要协同获取（锁序问题）？
- 是否存在跨线程状态依赖（一个线程的操作结果影响另一个线程的决策）？
- 并发模型是否可线性推理（如生产者-消费者）？还是存在非确定性交错？

无并发的模块此维度评为"低"。

示例推理：
> **引用证据**：3 个 mutex，分别保护 config_、cache_、stats_，无跨锁获取；使用条件变量做生产者-消费者通知。
> **主观评估**：三把锁职责独立，无锁序依赖；生产者-消费者模式成熟且可线性推理。并发结构清晰。
> **维度结论**：中

### 维度 D：认知复杂度

**核心问题**：向一个不熟悉该项目的开发者解释本模块，需要多少个核心概念？

量化参考：
- ≤ 3 个核心概念 → 低
- 4–6 个核心概念 → 中
- 7–10 个核心概念 → 高
- \> 10 个核心概念 → 极高

"核心概念"指无法跳过、必须理解才能修改模块的知识单元。例如：一个状态机模块可能需要理解"状态枚举、转移规则、事件触发机制"共 3 个概念。

示例推理：
> **引用证据**：模块实现协议栈的会话管理层，核心概念包括：(1) 会话生命周期状态机、(2) 超时重传机制、(3) 流控窗口算法、(4) 多路复用索引、(5) 错误恢复策略、(6) 与传输层的回调接口、(7) 会话迁移协议。
> **主观评估**：共 7 个核心概念，且概念间存在交叉依赖（如流控影响重传，重传影响状态机）。
> **维度结论**：高

### 维度 E：可拆分性评估

**核心问题**：模块内部是否存在清晰的拆分边界？

评估要点：
- 模块内是否存在多个可独立描述的功能职责？
- 是否存在互不相交的数据流路径？
- 内部子组件之间是否存在强双向耦合，还是可以单向分层？
- 若拆分，子模块间的接口是否自然（无需暴露大量内部状态）？

此维度不输出等级，而是输出结论：**可拆分** / **不宜拆分** / **需进一步评估**。

示例推理：
> **引用证据**：模块包含三个子目录 parser/、executor/、reporter/，分别负责解析、执行、报告生成。parser/ 输出 AST 供 executor/ 消费，executor/ 输出结果供 reporter/ 消费，数据流单向。
> **主观评估**：三个子功能职责清晰，数据流单向，拆分后子模块间接口自然（AST 结构和执行结果结构）。
> **维度结论**：可拆分（按功能职责拆为 parser、executor、reporter 三个子模块）

## 客观指标地板

客观指标作为等级的**最低下限**（地板），确保模型不会低估明显复杂的模块。模型可以基于结构化推理将等级评定为**高于**地板的等级，但不能低于地板。

| 触发条件 | 最低等级 | 例外说明 |
| -------- | -------- | -------- |
| 代码行数 > 10,000（有效行） | 高 | 若能证明代码为自动生成或高度重复模板（须提供具体证据，如"40 个函数结构相同，由代码生成器产生"），可申请豁免至原始推理结论 |
| 代码行数 > 5,000 或 类/结构体 > 10 | — | 无最低等级限制，但**必须**提供维度 E（可拆分性评估），不可跳过 |
| 涉及硬件交互、跨进程通信、或强非确定性（如随机+并发交叉） | 极高 | 无例外；硬件/跨进程的测试与验证成本本质性地高于纯软件模块 |

**地板规则的应用方式**：

1. 先完成 5 个维度的结构化推理，得出推理结论
2. 检查是否命中地板条件
3. 若推理结论低于地板，将最终等级提升至地板等级，并在报告中注明「因 [地板条件] 将等级从 [推理结论] 提升至 [地板等级]」
4. 若推理结论高于地板，直接采用推理结论

## 最终评级

### 综合方法

1. **汇总维度结论**：将维度 A–D 的四个等级结论列出
2. **应用地板检查**：检查是否命中客观指标地板条件
3. **综合判定**：最终等级不低于各维度结论中的最高等级，且不低于地板等级。若多个维度均为"高"，应考虑提升至"极高"

### 最终评级输出格式

```
最终等级：[低 / 中 / 高 / 极高]

评级依据（一段话）：
综合 5 个维度的评估结果，[简述主要复杂度来源和关键判断]。
[若命中地板条件] 客观指标地板：[条件描述]，将等级从 [推理结论] 提升至 [地板等级]。

维度汇总：
- A 代码规模：[等级]（[关键证据摘要]）
- B 依赖交互：[等级]（[关键证据摘要]）
- C 并发：[等级]（[关键证据摘要]）
- D 认知：[等级]（[关键证据摘要]）
- E 可拆分性：[结论]
```

### 与旧版机械规则的差异说明

若最终评级结果与旧版"升档/降档"机械规则可能给出的结论不同，**必须**在评级依据中额外说明差异原因。例如：

> 旧版规则下，12,000 行代码（升两档）+ I/O（升一档）会直接评为"极高"。本次评估基于结构化推理，判定代码规模维度为"中"（80% 为重复模板），I/O 维度为"低"（仅读取配置文件），综合评为"中"。差异原因：旧版规则以行数为主要驱动，未考虑信息密度。

### 反偷懒措施

P3 收敛评估将对模块评级进行交叉验证。若评级与模块报告中记录的客观指标、代码特征、数据流路径等存在明显矛盾（如报告记录了多个复杂数据流和并发不变量，但评级为"低"），收敛评估将标记为不通过并要求修正。

## 拆分评估触发条件

评级完成后，根据等级判断是否须进行拆分评估。拆分结论须在模块报告的「复杂度评级」一节中明确说明。

### 强烈建议拆分（等级 = 极高）

满足以下**任一**条件时，强烈建议将模块拆分为 2–4 个子模块：

| 触发条件 | 说明 |
|----------|------|
| 代码量 > 5000 行 | 单模块过大，静态分析成本高，下游路径追踪困难 |
| 内部存在 ≥2 个明显不同的功能职责 | 违反单一职责，难以隔离测试与审查 |
| 内部存在 ≥2 条完全独立的关键数据流路径 | 数据流不相交，说明模块可沿数据流边界拆分 |

**拆分目标**：每个子模块复杂度评级目标 ≤ 高，且子模块间依赖方向清晰（尽量单向）。

**拆分维度参考**（按优先级）：
1. 按**功能职责**：将负责不同业务功能的代码分入独立子模块
2. 按**数据流阶段**：将输入解析、核心处理、输出/事件派发分为不同子模块
3. 按**并发边界**：将需要加锁保护的共享状态与无锁的纯计算逻辑分离

### 须评估拆分（等级 = 高）

满足以下**全部**条件时，**须**评估是否可拆分，并在模块报告中填写拆分说明（已拆/不拆/待拆）：

| 触发条件 |
|----------|
| 代码量 > 3000 行 |
| 内部职责数 > 1（即模块承担了多个可独立描述的功能） |

若评估后决定**不拆分**，须在模块报告「复杂度评级」一节中写明**合格的不拆理由**（见下方「不拆理由格式要求」）。

### 不建议拆分（等级 = 中或低）

无论代码量多少，中/低复杂度模块通常不需要拆分，维持原有粒度即可。

### 拆分结论格式

拆分结论**必须**为以下三种之一，不允许使用「建议拆分」「建议进一步分析」等模糊表述作为终态：

| 结论 | 格式要求 | 后续动作 |
|------|---------|---------|
| **已拆** | `已拆（子模块：xxx_a, xxx_b, ...）` — 须列出子模块名称，且对应的子模块报告已生成 | 无（拆分已完成） |
| **不拆** | `不拆（理由：...）` — 须给出**合格的**具体技术理由（见下方理由格式要求） | 无（明确决策不拆） |
| **待拆** | `待拆（拆分方案：xxx_a, xxx_b, ...）` — 已识别拆分方案但尚未执行 | **此状态下 P3 收敛评估不可标记为「已收敛」**，须在下一轮 P1 迭代中执行子模块识别，完成后将结论更新为「已拆」 |

> **重要**：拆分说明为「待拆」的模块会阻断 P3 收敛评估通过（见 [收敛条件](convergence_criteria.md)）。Agent 必须通过 P1→P2 迭代将其推进为「已拆」或重新评估为「不拆」后，流程才能继续。

### 「不拆」理由格式要求

「不拆」理由**必须**包含以下至少一项具体技术论证：

- **(a) 拆分引入新依赖**：说明拆分后会新增哪些跨模块依赖或共享状态访问路径变化（如「ScenarioManager 与 ExecsManager 共享 running_topology_ 状态，拆分后需新增跨模块接口暴露该状态」）
- **(b) 内部强双向耦合**：指出模块内部哪些类/函数之间存在强双向调用关系，使得无法在不增加跨模块依赖的前提下划分边界（如「Parser 与 Validator 相互回调，拆分后两者需循环依赖」）
- **(c) 共享关键内部状态**：指出模块内多个子目录/子模块共享哪些关键内部数据结构或状态，拆分会导致该状态需要跨模块暴露（如「三个子模块共享内部缓存池 BufferPool，拆分后需将其提升为独立模块或公开内部接口」）

**不合格理由示例**（以下表述不被接受，分解审视将判定为不通过）：

| 不合格表述 | 原因 |
|-----------|------|
| "各子模块职责清晰" | 职责清晰恰恰说明拆分边界明确，是拆分可行的论据，不构成不拆理由 |
| "作为 XX 层需要保持完整性" | 过于笼统，未说明拆分的具体负面后果 |
| "通过接口解耦" | 接口解耦说明拆分成本低，不构成不拆理由 |
| "已有子目录划分" | 目录划分是代码组织方式，不等同于模块报告层面的拆分 |

## 在模块报告中的填写

在模块报告的「复杂度评级」一节中，根据报告层级填写不同详细程度的内容。L1 模板见 [module-report-L1.md](../templates/module-report-L1.md)，L2 模板见 [module-report-L2.md](../templates/module-report-L2.md)。

### L1 报告（标准模块报告）

```markdown
## 复杂度评级

- **等级**: [低 / 中 / 高 / 极高]
- **结构化推理摘要**:
  - A 代码规模: [等级]（[一句话关键证据]）
  - B 依赖交互: [等级]（[一句话关键证据]）
  - C 并发: [等级]（[一句话关键证据]）
  - D 认知: [等级]（[核心概念数量及列举]）
  - E 可拆分性: [可拆分 / 不宜拆分 / 需进一步评估]
  - 地板检查: [未命中 / 命中 XXX 条件，等级从 X 提升至 Y]
- **拆分说明**（等级为极高时必填，等级为高且满足拆分触发条件时**必填**）:
  - [已拆 / 不拆 / 待拆]（详见拆分评估触发条件）
```

| 指标 | 数值 | 采集命令 |
|------|------|---------|

| 类/结构体数量 | [数值] | [实际使用的命令] |
| 公开头文件数量 | [数值] | [实际使用的命令] |
| 内部依赖模块数 | [数值] | [模块列表] |
| 外部依赖类型 | [类型列表] | [检查结果] |
| 锁/互斥量数量 | [数值] | [实际使用的命令] |

### 结构化推理

**维度 A — 代码规模复杂度**
