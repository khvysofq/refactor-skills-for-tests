# 产出路径与报告结构约定

> **阅读时机**：Phase 04 或撰写/检查报告时；后续工作流按需引用时参阅「下游 Agent 使用约定」。

---

## 产出路径

所有产出均位于工程根目录下的 `docs/codearch/`（相对仓库根）。

| 产出             | 路径                                     | 说明                                                                     |
| ---------------- | ---------------------------------------- | ------------------------------------------------------------------------ |
| 总体报告         | `docs/codearch/overall_report.md`        | 工程概览、模块列表（含链接）、构建/测试摘要                              |
| 模块报告         | `docs/codearch/modules/<module_name>.md` | 单模块报告，`<module_name>` 为模块标识（如目录名或 target 名，不含空格） |
| 子模块报告       | `docs/codearch/modules/<parent>_<submodule>.md` | 深入分析产生的子模块报告，命名为父模块名\_子模块名（平铺命名，不使用目录嵌套）；多级递归时继续平铺，如 `<grandparent>_<parent>_<child>.md` |
| 构建与测试       | `docs/codearch/build_and_tests.md`       | 编译系统、单元测试体系、运行测试方式                                     |
| 模块地图（可选） | `docs/codearch/module_map.md`            | 模块列表与依赖图汇总，可与总体报告合并或独立                             |

## 总体报告必须包含的章节

1. **工程名称与分析日期**（文档头部）
   - **分析范围/版本**（可选但建议）：仓库 commit/tag、分析范围（如全库或子路径），供下游 Agent 判断报告是否仍适用于当前代码。
2. **工程概览**
   - 工程目标（主要做什么）
   - 输入（配置、文件、命令行、网络等）
   - 输出（结果文件、服务响应、日志等）
   - 主流程（步骤列表或流程图）
3. **技术特征概览**（为下游风险分析提供全局视图）
   - 语言与标准（主要语言、语言标准、主要依赖库）
   - 技术特征统计（各技术特征涉及的模块数和主要模块）
   - 入口点索引（main() 位置、网络入口、配置入口等）
4. **模块列表**
   - 每个模块一行或表格行，包含：模块名、**路径/范围**（该模块对应的目录或 glob，如 `src/core/`、`src/parser/*.cpp`）、复杂度等级、**指向该模块报告的链接**
   - 路径/范围便于「代码路径 → 模块 → 报告」的按需加载。
   - 链接格式：相对路径 `modules/<module_name>.md`，例如 `[core/utils](modules/core_utils.md)`
   - 可选：模块可含固定键（如 `role: lib|service|driver`），便于按角色过滤而无需打开每份报告。
5. **模块依赖图**（必填）
   - 以 Mermaid 图形式展示所有模块间的依赖方向，提供全局拓扑视图，下游 Agent 无需逐一加载每份模块报告即可理解全局结构。
   - **每条依赖边须标注依赖类型**，取值从以下枚举中选取：
     - `调用`：A 直接调用 B 的函数/方法（最常见）
     - `数据传递`：A 将数据对象（含所有权）传给 B，或 B 产生数据交给 A
     - `事件通知`：A 注册回调或监听器，B 触发事件/回调
     - `配置读取`：A 从 B 读取配置信息
     - `继承`：A 继承/实现 B 定义的接口或基类
   - **枢纽模块标注**：被 ≥3 个其他模块依赖的模块，在图中用注释或标注标明（如 `[HUB]`），提示后续审查优先关注。
   - **跨模块所有权转移标注**：若某条依赖边涉及资源/对象所有权在模块间转移，须在该边旁注明（如 `[所有权转移]`），供风险分析时重点关注内存安全。
6. **构建与测试摘要**
   - 简要说明或直接链接到 `build_and_tests.md`
7. **后续分析建议**（可选）

## 模块报告必须包含的章节

1. **模块名与路径**（文档头部）
2. **职责描述**
3. **边界**
   - 输入
   - 输出
4. **依赖**
   - 内部依赖（其他模块）
   - 外部依赖（系统调用、库、I/O 等）
5. **复杂度评级**
   - 等级：低 / 中 / 高 / 极高
   - 可选：简要依据
6. **关键设计要点**（建议必填）
   - 3–5 条要点，如：主要入口函数/类、核心数据结构、线程/并发假设、错误处理策略、扩展点等；供后续代码审查与重构 Agent 快速建立上下文。
7. **代码特征**（必填，为下游风险分析提供索引）
   - 内存管理：方式（RAII/智能指针/手动/混合）+ 简要说明
   - 并发模型：是否多线程 + 同步机制 + 简要说明
   - I/O 操作：文件 I/O、网络 I/O（是/否）+ 简要说明
   - 外部数据处理：是否处理外部输入 + 数据来源 + 简要说明
   - 错误处理：策略（返回错误码/异常/混合）+ 简要说明
8. **关键代码位置索引**（必填）
   - 主要入口点（函数/类、位置、说明）
   - 外部数据入口（若有）：含首道校验位置
   - 错误/异常路径（关键）：触发条件、清理逻辑位置、是否完整
9. **关键数据流路径**（必填）
   - 模块内关键的数据从入口到出口的流转路径
   - 每条路径含：起点、经过、终点、涉及的所有权转移
   - 为下游风险分析提供路径追踪基础
10. **生命周期与所有权模型**（必填，涉及手动内存管理或跨函数/跨模块生命周期时）
    - 关键对象/资源的创建位置、销毁位置、所有权模型（RAII / 调用方释放 / 框架释放 / 共享所有权）、转移点
    - 无动态分配模块可标注「无需记录」
11. **并发不变量**（仅多线程模块必填）
    - 受保护数据/结构、保护锁/机制、锁获取顺序约定、线程上下文假设
    - 非多线程模块标注「本模块为单线程模型」
12. **与其它模块的关系**（必填，结构化表格）
    - **上游依赖**（本模块依赖的模块）：模块名、依赖类型（调用/数据传递/事件通知/配置读取/继承）、是否涉及跨模块所有权转移、简要说明
    - **下游依赖**（依赖本模块的模块）：模块名、依赖类型、是否涉及跨模块所有权转移、简要说明
    - 依赖类型枚举与总体报告「模块依赖图」保持一致，便于自动对照验证。
    - 若本模块为枢纽模块（被 ≥3 个模块依赖），须在此节顶部注明「本模块为枢纽模块」。
13. **子模块索引**（仅已做深入分析的父模块必填）
    - 深入分析日期/轮次
    - 拆分依据（按功能职责 / 按数据流阶段 / 按并发边界）
    - 子模块列表：子模块名、路径/范围、复杂度、报告链接
    - 说明：原有报告内容作为顶层概览保留，子模块报告提供更细粒度的分析

不包含可测试性状态（如 S1–S4）、测试策略等重构专用字段，保持纯架构分析。代码特征只记录客观事实，不做风险判断。

## 引用格式

- 在总体报告中引用模块报告：使用 Markdown 链接 `[模块名](modules/<module_name>.md)`。
- 模块名中的路径分隔符建议统一为下划线或短横线（如 `core_utils` 或 `core-utils`），与文件名一致，便于后续 Agent 按需加载单份文件。

## 下游 Agent 使用约定

供后续工作流（代码审查、风险分析、重构、写测试等）在按需引用本产出时使用：

- **何时只读总体报告**：建立全局印象、选择模块、判断分析范围/版本是否仍适用（结合头部分析范围/版本与当前仓库 commit）。
- **何时使用模块依赖图**：风险分析时，先通过总体报告的「模块依赖图」快速定位枢纽模块（标注 `[HUB]`）和跨模块所有权转移边（标注 `[所有权转移]`），优先对这些高风险拓扑位置做深度审查，而无需打开每份模块报告。
- **何时使用技术特征概览**：风险分析时，根据「技术特征统计」快速定位涉及特定技术（如多线程、外部输入处理）的模块，确定分析优先级。
- **何时按需加载模块报告**：分析、审查或重构涉及某模块时；根据总体报告中「模块列表」的 **路径/范围** 列，由当前文件路径定位应加载的 `modules/<name>.md`。
- **如何使用模块间关系表**：加载某模块报告后，通过「与其它模块的关系」的结构化表格快速理解该模块的上下游：重点关注依赖类型为「数据传递」且标注「跨模块所有权转移=是」的条目，这些边是内存安全审查的优先入口。
- **如何使用代码特征**：风险分析 Agent 可根据模块报告的「代码特征」章节，快速判断该模块是否涉及特定风险点（如手动内存管理、外部输入处理），决定是否深入审查。
- **如何使用数据流路径**：风险分析 Agent 在做深度审查时，应沿模块报告的「关键数据流路径」对实际代码做路径追踪，而非仅做模式匹配扫描。每条数据流路径标注了所有权转移点，Agent 应在这些转移点验证调用方与被调用方的所有权语义是否一致。
- **如何使用生命周期与所有权模型**：对涉及手动内存管理的模块，Agent 应逐一检查「生命周期与所有权模型」中列出的每个对象/资源，确认所有执行路径（含错误路径）上创建-销毁配对完整。
- **如何使用并发不变量**：并发竞态审查时，Agent 应以「并发不变量」表为基准，检查代码中对受保护数据的每次访问是否都在指定锁的保护下，以及锁获取顺序是否与约定一致。
- **推荐用法**：进行风险分析或深度审查时，先加载 `overall_report.md` 查看模块依赖图与技术特征概览（定位枢纽模块、所有权转移边、高风险特征模块），再根据改动涉及的文件/路径加载对应的 `modules/*.md`；通过「与其它模块的关系」表理解上下游所有权语义，利用「代码特征」和「关键代码位置索引」快速定位需要关注的代码区域，然后基于「关键数据流路径」「生命周期与所有权模型」「并发不变量」做路径级深度追踪。

## Agent 解析约定

- **固定标题**：总体报告与模块报告的必须章节使用固定标题（如 `## 工程概览`、`## 关键设计要点`、`## 职责描述`），便于 Agent 按标题解析。
- **YAML frontmatter**（可选）：允许在报告头部使用 YAML frontmatter 提供摘要字段。
  - 总体报告：如 `summary`、`scope_commit`、`module_count`，供 Agent 做轻量解析或 embedding。
  - 模块报告：如 `summary`、`complexity`、`path`，便于先扫再决定是否读全文。
  - 格式与用途在模板中简要说明即可。
