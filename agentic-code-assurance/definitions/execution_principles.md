# 通用执行原则

> **阅读时机**：首次执行本工作流时通读；后续执行时按需回顾。本文档定义所有阶段共享的执行原则。

---

## 原则列表

1. **最小上下文原则**：仅加载当前步骤需要的文档；总体报告仅列模块链接，不内嵌模块全文，便于按需加载单份模块报告。Phase 02 按模块或风险维度分批，每批只加载对应模块报告，不一次性加载全部。验证与修复时按任务逐条或分批处理，单次仅加载当前任务相关模块报告与源码。

2. **验收驱动**：每个 Phase 与 Skill 执行完必须验证验收标准。

3. **状态持久化**：产出统一放在约定目录（`docs/codearch/`、`docs/risk_tasks/`、`docs/remediation/`），路径与结构见各阶段的产出结构定义文档，便于中断后恢复。

4. **按决策树进入**：从 Q1 开始，首个「否」进入对应 Phase，避免跳过或重复执行。

5. **反馈及时**：执行中一旦发现工程理解文档与代码不符，按 [反馈操作约定](feedback_protocol.md) 更新并可选记录。

6. **阶段衔接**：前置阶段产出未就绪时，先执行根 Workflow 中对应阶段，再进入当前阶段决策树。

7. **产出路径固定**：各阶段产出遵循对应的产出结构定义文档（`output_structure.md`、`task_output_structure.md`、`remediation_output_structure.md`）。

8. **文档优先**：优先利用项目已有文档（README、测试、示例）作为信息来源，文档内容优先于代码推断。

9. **深度验证**：对高复杂度模块（「高」或「极高」）通过运行测试进行验证，确保理解正确；验证需要构建环境时可提前执行 Phase 03。

10. **来源标注**：使用示例需标注来源（文件路径或「Agent 编写」），便于追溯和验证。
