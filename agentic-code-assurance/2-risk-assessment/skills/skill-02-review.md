# Skill 02: 深度审查

> **触发条件**：Phase 02 指示（入口决策 Q2 为「否」）  
> **目标**：对分析范围内的模块按风险维度进行深度代码审查，识别疑似 BUG 并写入任务列表（或临时清单），每条须含必填字段。

---

## 输入

- 分析范围：`docs/risk_tasks/scope.md` 或 task_list 头部（选定模块、风险维度）
- `docs/codearch/overall_report.md`
- **按需加载**的 `docs/codearch/modules/<module_name>.md`（仅加载当前批审查涉及的模块，不要一次性加载全部）
- 仓库源码

---

## 输出

| 产出 | 路径 | 必须 |
|------|------|------|
| 疑似 BUG 条目 | `docs/risk_tasks/task_list.md` 或临时文件 | ✓（可由 Phase 03 合并/去重后定稿） |

每条须符合 [task_output_structure](../definitions/task_output_structure.md)：位置、简要描述、风险类型、关联模块；可选置信度、建议验证方式、复现思路。

---

## 审查深度等级

根据模块复杂度确定审查深度，集中资源到高风险区域：

| 模块复杂度 | 审查深度 | 方法                                                                                                |
| ---------- | -------- | --------------------------------------------------------------------------------------------------- |
| 极高       | L3-逐路径追踪 | 对每个关键函数，枚举所有执行路径（正常/错误/异常），逐路径验证资源管理与并发安全              |
| 高         | L2-关键路径追踪 | 对关键入口函数，追踪主路径和主要错误路径，验证所有权转移与资源配对                           |
| 中         | L1-模式扫描+抽样 | 按审查模式做模式扫描，对可疑处做局部路径分析                                                 |
| 低         | L0-模式扫描 | 仅按审查模式做模式匹配，快速扫描明显问题                                                       |

---

## 核心任务

### 任务 1: 模块内路径追踪审查

1. **按模块或按风险维度分批**，每批仅加载当前批对应的 `docs/codearch/modules/<module_name>.md`。
2. **确定审查深度**：根据上方「审查深度等级」表和模块复杂度，确定本模块的审查方法。
3. **加载语义上下文**：不仅读取「代码特征」与「关键代码位置索引」，还须加载：
   - 「关键数据流路径」：沿每条数据流路径追踪实际代码
   - 「生命周期与所有权模型」：验证每个关键对象的所有权语义
   - 「并发不变量」（多线程模块）：以此为基准检查并发安全
4. **按审查模式执行路径追踪**：
   - 加载 [审查模式定义](../definitions/review_patterns.md)，选择与当前风险维度对应的模式
   - 对 L2/L3 深度的模块：沿每条数据流路径，逐步应用审查模式，追踪代码而非仅 grep 关键词
   - 对 L0/L1 深度的模块：按审查模式做关键词扫描，对可疑处做局部路径验证
5. **记录疑似问题**：每条疑似 BUG 按 [task_output_structure](../definitions/task_output_structure.md) 记录所有必填字段（含推理依据、已排除的保护机制、需验证的前提假设），并标注对应的审查模式编号（如 M-1、C-2）。**若审查过程中因模块报告信息不足（如所有权模型未记录、数据流路径不完整、模块边界模糊）导致推理受阻，须在该任务条目的 `knowledge_gap` 字段记录缺失的具体信息**，供后续知识库补强使用（见 [反馈操作约定](../../definitions/feedback_protocol.md)）。
6. 将条目写入 `docs/risk_tasks/task_list.md`（可追加）或临时文件，供 Phase 03 汇总。

**最小上下文原则**：不要一次性加载全部模块报告；每批只读当前审查模块的报告与对应源码。

### 任务 1.5: 跨模块交互审查

1. 从 `docs/codearch/overall_report.md` 的模块依赖图中识别**关键的模块交互边界**（特别是高复杂度模块之间的调用关系）。
2. 对每条跨模块调用路径，按需加载两端模块的报告，审查以下方面：
   - **所有权转移**：对象/资源在模块边界处的所有权是否在调用方和被调用方之间一致？
   - **错误传播**：错误码/异常是否能完整地跨模块边界传播？是否有中间层吞掉错误？
   - **线程上下文**：调用方的线程上下文假设与被调用方是否一致？（如：调用方认为在主线程，被调用方实际在回调线程中执行）
3. **重点审查「回调跨越模块边界」**的场景：
   - 模块 A 注册回调给模块 B，回调触发时引用的模块 A 对象是否仍存活？
   - 回调在哪个线程上下文中执行？是否与注册时的假设一致？
4. 将跨模块问题同样按 [task_output_structure](../definitions/task_output_structure.md) 记录。

### 任务 2: 验收自检

- 确认分析范围内所有选定模块/维度已完成审查。
- 确认高复杂度模块使用了 L2/L3 路径追踪审查（而非仅 L0 模式匹配）。
- 确认跨模块交互审查已覆盖关键的模块边界。
- 确认每条条目含所有必填字段（含推理依据）；执行 Phase 02 中的验收检查（若已写 task_list）。
