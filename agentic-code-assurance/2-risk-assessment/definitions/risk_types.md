# 风险类型定义

> **阅读时机**：Phase 01 定策略、Phase 02 做审查、撰写任务条目时。任务列表中的「风险类型」字段取值须与本定义一致。

---

## 与阶段一「代码特征」的对应关系

阶段一（代码认知）在模块报告中记录的「代码特征」为客观事实（如「使用 malloc/free」「涉及多线程」）；本阶段的风险类型用于对**疑似 BUG** 做分类判断，与代码特征的维度对齐，便于从知识库按特征筛模块后再按类型记任务。

| 代码特征维度（阶段一） | 风险类型（本阶段） | 典型问题 |
|------------------------|--------------------|----------|
| 内存管理               | 内存管理           | double-free、use-after-free、泄漏、缓冲区溢出、未初始化变量、不安全类型转换 |
| 并发模型               | 并发竞态           | 数据竞争、死锁、未同步访问共享变量 |
| I/O 操作、外部数据处理 | I/O 与外部输入     | 未校验长度/格式、路径遍历、注入、格式串注入、栈缓冲区溢出 |
| 错误处理               | 错误处理           | 未检查返回值、异常未处理、错误路径泄漏资源 |
| 循环与热路径、数据传递方式 | 性能           | 循环内昂贵操作、不必要的全量数据拷贝、算法复杂度不当 |
| 代码结构、模块设计     | 可维护性           | 魔法数字与硬编码、过长函数与高圈复杂度、设计原则违反 |
| 平台依赖、数据表示     | 可移植性           | 平台特定 API 调用、字节序假设、数据类型大小假设 |

## 风险类型枚举

任务条目的「风险类型」字段须从以下类型中选取（可后续扩展）：

| 类型 | 说明 | 审查模式 |
|------|------|----------|
| **内存管理** | 与动态内存、缓冲区、生命周期相关 | M-1 ~ M-7：分配释放配对、所有权转移、缓冲区边界、整数溢出、错误路径清理、未初始化变量、不安全类型转换 |
| **并发竞态** | 多线程/多任务下的共享数据与同步 | C-1 ~ C-4：锁顺序一致性、检查-执行竞态、回调生命周期、无保护共享状态 |
| **I/O 与外部输入** | 来自文件/网络/命令行/配置的数据处理 | IO-1 ~ IO-5：信任边界、长度格式校验、解析器状态机、格式串注入、栈缓冲区溢出 |
| **错误处理** | 返回值、异常、错误路径 | E-1 ~ E-3：返回值传播、错误路径资源释放、异常安全保证 |
| **性能** | 运行时效率问题 | PERF-1 ~ PERF-3：循环内昂贵操作、不必要数据拷贝、算法复杂度不当 |
| **可维护性** | 代码质量与长期可维护性 | MAINT-1 ~ MAINT-3：魔法数字与硬编码、过长函数与高圈复杂度、设计原则违反 |
| **可移植性** | 跨平台兼容性 | PORT-1 ~ PORT-2：平台特定 API 调用、字节序与数据类型假设 |
| **其它** | 难以归入上述类型的逻辑或实现问题 | 逻辑错误、单位混用、配置错误等 |

## 影响等级

任务条目的「影响等级」字段须从以下等级中选取，用于指导修复优先级排序：

| 等级 | 定义 | 判断标准 |
|------|------|----------|
| **Critical** | 可直接导致安全漏洞、数据丢失或系统崩溃 | 可被外部触发且无需特殊条件；影响核心功能或数据完整性；涉及内存破坏或任意代码执行 |
| **High** | 高概率触发的正确性问题或资源泄漏 | 在常规使用路径上可触发；可能导致功能异常或服务降级；需要特定但常见的条件 |
| **Medium** | 存在风险但触发条件较苛刻，或影响范围有限 | 需要较特殊的输入或运行条件；影响局部功能或非核心路径；存在部分缓解措施 |
| **Low** | 代码质量问题或极端边界条件下的缺陷 | 极难触发或仅在理论上存在风险；影响可忽略或有完整的防护；属于最佳实践偏离而非功能缺陷 |

**等级选取原则**：

- 当 BUG 可被外部不可信输入直接触发时，至少为 **High**。
- 涉及内存破坏（溢出、use-after-free）的问题，至少为 **High**，若可远程触发则为 **Critical**。
- 性能和可维护性类问题通常为 **Medium** 或 **Low**，除非直接影响系统可用性。
- 可移植性问题在目标平台单一时通常为 **Low**；跨平台部署场景下可升至 **Medium** 或 **High**。

## 使用说明

- Phase 01 确定分析范围时，可结合 overall_report 的「技术特征概览」选择要重点审查的风险类型与模块。
- Phase 02 深度审查时，按选定模块的「代码特征」定位代码，再按上述类型对每条疑似 BUG 标注「风险类型」，并评估「影响等级」。
- 撰写任务列表时，每条条目的「风险类型」和「影响等级」必须从本文档中的枚举值选取，以便阶段三优先级排序与后续统计一致。
