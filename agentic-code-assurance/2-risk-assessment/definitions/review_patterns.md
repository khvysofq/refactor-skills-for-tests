# 审查模式定义

> **阅读时机**：Phase 02 做深度审查时。本文档定义每种风险类型的具体审查模式（Review Pattern），要求 Agent 沿代码路径做语义追踪，而非仅做关键词模式匹配。

---

## 使用说明

- 每条审查模式定义了「追踪什么」「在每步验证什么」「什么构成真正的问题 vs. 误报」。
- Agent 在执行 [Skill 02: 深度审查](../skills/skill-02-review.md) 时，须按照目标模块涉及的风险类型，逐条应用对应的审查模式。
- 审查时应结合模块报告中的「关键数据流路径」「生命周期与所有权模型」「并发不变量」等语义信息做路径追踪，而非仅靠 grep 扫描。
- 每条疑似问题须标注其对应的审查模式编号（如 M-1、C-2），便于追溯与验证。

---

## 一、内存管理审查模式

### Pattern M-1: 分配-释放配对追踪

**追踪目标**：每个 malloc/calloc/realloc/new 调用

**追踪路径**：
1. 起点：分配调用位置
2. 路径：枚举从分配点到函数结束的**所有可能执行路径**（包括正常返回、early return、goto、错误处理分支、异常）
3. 终点：对应的 free/delete 调用

**每步验证**：
- 每条执行路径上是否都有对应的释放？
- 是否存在某条错误路径遗漏了释放？
- 释放前是否有重复释放的可能（double-free）？

**确认问题**：至少一条执行路径上缺少释放，且该路径在运行时可达
**排除误报**：
- 对象所有权已转移给调用方或框架（查阅模块报告「生命周期与所有权模型」确认）
- 对象由 RAII 包装器管理（如 unique_ptr）
- 释放在析构函数中统一处理且析构一定被调用

### Pattern M-2: 所有权转移验证

**追踪目标**：模块报告「生命周期与所有权模型」中列出的每个对象/资源

**追踪路径**：
1. 起点：对象创建位置
2. 路径：对象在函数间传递的每个转移点
3. 终点：对象销毁位置

**每步验证**：
- 在每个转移点，调用方和被调用方对「谁负责释放」的理解是否一致？
- 函数签名（裸指针 vs. unique_ptr vs. shared_ptr）是否清晰表达所有权语义？
- 回调函数中引用的对象在回调触发时是否仍然存活？

**确认问题**：调用方和被调用方对所有权有不一致的理解，且可能导致 double-free 或 use-after-free
**排除误报**：
- 有明确的文档/注释约定所有权语义
- 框架设计保证了对象生命周期覆盖回调执行期

### Pattern M-3: 缓冲区边界追踪

**追踪目标**：每个 memcpy/memmove/memset 调用，以及数组下标访问

**追踪路径**：
1. 起点：size/length 参数的来源（变量定义或外部输入）
2. 路径：size 参数从来源到使用点的所有变换（算术运算、类型转换）
3. 终点：memcpy 等函数调用或数组访问

**每步验证**：
- size 参数是否来自外部输入（网络/文件/用户输入）？
- 从来源到使用点之间是否有长度校验？校验是否在所有路径上都执行？
- 算术运算是否可能整数溢出（特别是乘法：`count * sizeof(element)`）？
- 目标缓冲区大小是否 ≥ 拷贝长度？

**确认问题**：size 来自外部输入且到达 memcpy 时未经充分校验，或算术运算存在溢出风险
**排除误报**：
- size 在到达使用点前已有上限检查
- size 来源为内部可信数据（非外部输入）
- 有编译时或运行时的边界保护（如 `strlcpy`、`snprintf`）

### Pattern M-4: 整数溢出导致分配异常

**追踪目标**：动态分配中的 size 计算表达式（如 `malloc(n * sizeof(T))`）

**追踪路径**：
1. 起点：参与 size 计算的变量
2. 路径：变量的来源与变换
3. 终点：分配函数调用

**每步验证**：
- 参与乘法/加法的变量是否有上限？
- 计算结果是否可能超过 `SIZE_MAX`？
- 分配后是否检查了返回值（NULL 检查）？

**确认问题**：size 计算中存在无保护的算术运算，且输入值可由外部控制
**排除误报**：输入值有明确的上限约束，或计算在编译时已知

### Pattern M-5: 错误路径资源清理

**追踪目标**：函数内分配的所有资源（内存、文件句柄、锁等）

**追踪路径**：
1. 起点：函数入口（标记所有可能分配资源的语句）
2. 路径：从每个资源分配点到每个错误返回点
3. 终点：错误返回语句

**每步验证**：
- 错误返回前是否释放了所有已分配的资源？
- 多个资源的释放顺序是否正确（后分配先释放）？
- 是否使用了 goto cleanup 模式或 RAII 保证清理？

**确认问题**：某个错误路径上有资源未被释放
**排除误报**：使用了 RAII 或 goto cleanup 模式统一管理

### Pattern M-6: 未初始化变量使用

**追踪目标**：局部变量声明（特别是指针、数组、结构体）

**追踪路径**：
1. 起点：变量声明位置（无显式初始化）
2. 路径：从声明到首次读取之间的所有可能执行路径
3. 终点：变量首次被读取（作为右值使用）的位置

**每步验证**：
- 声明时是否有显式初始化？
- 从声明到读取之间，是否所有路径都经过赋值/初始化？
- 是否存在某条分支路径跳过了赋值直接读取？
- 对结构体/类对象，成员是否在构造函数中全部初始化？

**确认问题**：存在至少一条执行路径，变量在未初始化状态下被读取
**排除误报**：
- 编译器保证零初始化（如全局变量、static 变量）
- 变量在声明后立即被 memset/bzero 清零
- 变量为类类型且构造函数保证所有成员初始化

### Pattern M-7: 不安全类型转换

**追踪目标**：显式和隐式类型转换（`(type)expr`、`reinterpret_cast`、`static_cast`、隐式窄化转换）

**追踪路径**：
1. 起点：类型转换表达式位置
2. 路径：转换结果被使用的所有位置
3. 终点：转换后的值影响控制流或内存操作的位置

**每步验证**：
- `void*` 转换回具体类型时，类型是否与原始类型一致？
- 有符号/无符号混用时，负值是否可能被解释为大正数？
- 窄化转换（如 `int64_t → int32_t`）是否可能丢失数据？
- `reinterpret_cast` 是否违反严格别名规则（strict aliasing）？

**确认问题**：类型转换可能导致数据截断、类型混淆或未定义行为，且转换后的值影响关键逻辑
**排除误报**：
- 转换前有范围检查确保不会溢出/截断
- `void*` 的类型信息通过标签或上下文保证正确
- 转换在编译时已知安全（如 `uint8_t → int`）

## 二、并发竞态审查模式

### Pattern C-1: 锁顺序一致性

**追踪目标**：模块报告「并发不变量」中列出的所有锁

**追踪路径**：
1. 起点：每个锁获取操作（lock/pthread_mutex_lock）
2. 路径：在同一函数或调用链中获取的其他锁
3. 终点：锁释放

**每步验证**：
- 同一代码路径中是否持有多把锁？
- 多把锁的获取顺序是否与「并发不变量」中约定一致？
- 是否存在某条路径以 A→B 顺序获取，另一条路径以 B→A 顺序获取？

**确认问题**：存在两条代码路径以不同顺序获取同一组锁（死锁风险）
**排除误报**：
- 两条路径不可能被并发执行
- 使用了 try_lock 等非阻塞获取方式

### Pattern C-2: 检查-执行竞态（Check-Then-Act）

**追踪目标**：条件检查与后续操作之间的间隙

**追踪路径**：
1. 起点：对共享变量的条件检查（if/while）
2. 路径：从检查到基于检查结果的操作
3. 终点：操作执行

**每步验证**：
- 检查与操作之间是否在同一锁保护下？
- 是否有其他线程可能在检查与操作之间修改该变量？
- 若使用双重检查模式（double-checked locking），第二次检查是否在锁内？

**确认问题**：检查与操作不在同一原子操作/锁保护范围内，且共享变量可被并发修改
**排除误报**：
- 检查与操作在同一锁的保护下
- 变量为 thread-local 或仅单线程访问
- 使用了原子操作（atomic compare-and-swap）

### Pattern C-3: 回调生命周期安全

**追踪目标**：注册回调函数时引用的对象

**追踪路径**：
1. 起点：回调注册位置（对象被捕获/引用）
2. 路径：从注册到回调实际触发之间的时间窗口
3. 终点：回调触发与执行

**每步验证**：
- 回调触发时，其引用的对象是否仍然存活？
- 对象是否可能在回调触发前被释放（特别是 `delete this` 模式）？
- 回调执行在哪个线程上下文中？该线程是否有权访问对象？

**确认问题**：回调可能在对象已被释放后触发（use-after-free），或在错误的线程上下文中访问对象
**排除误报**：
- 框架保证对象生命周期覆盖回调执行期
- 使用了 shared_ptr/weak_ptr 管理对象生命周期
- 有显式的取消注册机制在对象销毁前调用

### Pattern C-4: 无保护共享状态访问

**追踪目标**：模块报告「并发不变量」中列出的受保护数据

**追踪路径**：
1. 起点：共享数据的声明/定义
2. 路径：所有读写该数据的代码位置
3. 终点：每个访问点

**每步验证**：
- 每次访问（读或写）是否在约定的锁保护下？
- 是否存在某个访问点遗漏了锁保护？
- 对于标注为 atomic 的变量，操作是否使用了正确的内存序？

**确认问题**：存在未被锁保护的共享数据访问，且该访问可被并发执行
**排除误报**：
- 该访问只在初始化阶段执行（此时无并发）
- 使用了其他同步机制（如 eventfd、管道通知）
- 变量为只读（初始化后不再修改）

## 三、I/O 与外部输入审查模式

### Pattern IO-1: 信任边界追踪

**追踪目标**：模块报告「关键代码位置索引 - 外部数据入口」中的每个入口

**追踪路径**：
1. 起点：外部数据进入系统的位置（recv/read/fread/解析函数入口）
2. 路径：数据从入口经过的所有处理步骤
3. 终点：数据被用于内存操作（memcpy）、控制流（if/switch）或输出（send/write）

**每步验证**：
- 数据在被使用前是否经过了校验（长度、格式、范围）？
- 校验是否在所有到达使用点的路径上都执行？
- 校验后数据是否可能被再次修改（TOCTOU 问题）？

**确认问题**：外部数据到达使用点时未经充分校验
**排除误报**：
- 校验在上层协议/框架中已完成
- 数据使用方式不依赖于数据内容（如仅做日志记录）

### Pattern IO-2: 长度/格式校验完整性

**追踪目标**：协议解析器中的长度字段和格式字段

**追踪路径**：
1. 起点：从网络/文件读取的长度/大小字段
2. 路径：长度字段用于内存分配或缓冲区访问的路径
3. 终点：malloc(length) 或 memcpy(buf, src, length) 等调用

**每步验证**：
- 长度字段是否有上限检查？上限值是否合理？
- 长度字段是否有下限检查（非负/非零）？
- 长度字段是否用于计算（可能溢出）？
- 格式字段的取值范围是否被验证？

**确认问题**：长度字段来自外部输入且到达内存操作时无合理上限检查
**排除误报**：
- 有全局的最大消息/缓冲区限制
- 长度字段在解析框架层已被校验

### Pattern IO-3: 解析器状态机正确性

**追踪目标**：协议解析器的状态转换逻辑

**追踪路径**：
1. 起点：解析器初始状态
2. 路径：所有可能的状态转换路径
3. 终点：解析完成或错误状态

**每步验证**：
- 每个状态是否处理了所有可能的输入？
- 无效输入是否能使解析器进入错误状态（而非未定义行为）？
- 错误状态是否能正确清理已分配的资源？

**确认问题**：某个状态下的特定输入导致未定义行为或资源泄漏
**排除误报**：有 default 分支处理未知输入并进入错误状态

### Pattern IO-4: 格式化字符串注入

**追踪目标**：printf/sprintf/fprintf/syslog/snprintf 系列函数的格式串参数

**追踪路径**：
1. 起点：格式化输出函数的调用位置
2. 路径：格式串参数的来源追踪（回溯到定义/赋值位置）
3. 终点：格式串参数的最初来源

**每步验证**：
- 格式串参数是否为编译时常量字符串？
- 若格式串来自变量，该变量是否可能被外部输入影响？
- 用户数据是否被直接作为格式串参数（如 `printf(user_input)` 而非 `printf("%s", user_input)`）？
- syslog/日志函数是否同样存在格式串注入风险？

**确认问题**：格式串参数来自外部输入或受外部输入影响，攻击者可利用 `%n`/`%x` 等格式符读写内存
**排除误报**：
- 格式串为编译时常量
- 外部输入仅作为格式化参数（如 `printf("%s", user_data)`）
- 使用了安全的日志框架，框架内部保证格式串安全

### Pattern IO-5: 栈缓冲区溢出

**追踪目标**：对栈上缓冲区的写入操作，特别是 `strcpy`、`strcat`、`sprintf`、`gets`、`scanf("%s", ...)` 等无边界限制的函数

**追踪路径**：
1. 起点：栈上缓冲区声明（如 `char buf[256]`）
2. 路径：缓冲区被写入的所有位置
3. 终点：写入操作的调用点

**每步验证**：
- 是否使用了无边界限制的函数（`strcpy`、`sprintf`、`gets`）？
- 若使用有边界的版本（`strncpy`、`snprintf`），边界值是否正确（不超过缓冲区大小）？
- 写入数据的来源是否可被外部控制？
- 多次写入是否会累积超过缓冲区大小？

**确认问题**：外部可控数据通过无边界限制（或边界限制不正确）的函数写入栈缓冲区，可能导致栈溢出
**排除误报**：
- 写入数据大小在编译时可确定且不超过缓冲区
- 使用了安全函数且边界值正确
- 输入已在上游被截断到安全长度

## 四、错误处理审查模式

### Pattern E-1: 返回值传播完整性

**追踪目标**：可能失败的函数调用（系统调用、内存分配、I/O 操作）

**追踪路径**：
1. 起点：可失败函数的调用点
2. 路径：返回值被检查和传播的路径
3. 终点：错误信息到达调用方或用户

**每步验证**：
- 返回值是否被检查（而非忽略）？
- 检查到错误后是否传播给上层调用方？
- 错误传播链是否完整（中间是否有吞掉错误的层）？

**确认问题**：返回值未被检查且函数失败可能导致后续操作在无效状态上执行
**排除误报**：
- 返回值被故意忽略（有注释说明原因）
- 函数失败不影响后续逻辑的正确性
- 使用了 `(void)func()` 显式忽略

### Pattern E-2: 错误路径资源释放

**追踪目标**：函数中获取的资源（内存、文件句柄、锁、网络连接）

**追踪路径**：
1. 起点：资源获取位置
2. 路径：从获取到每个错误返回点
3. 终点：错误返回语句

**每步验证**：（与 M-5 相同，此处强调错误处理维度）
- 错误返回前是否释放了已获取的全部资源？
- 部分初始化状态下的错误返回是否只释放了已获取的资源（不释放未获取的）？

**确认问题**：错误路径上存在资源泄漏
**排除误报**：使用了 RAII 或统一的错误处理/清理机制

### Pattern E-3: 异常安全保证

**追踪目标**：C++ 代码中可能抛出异常的操作（new、STL 操作、用户代码）

**追踪路径**：
1. 起点：可能抛异常的操作
2. 路径：异常传播路径上的已获取资源
3. 终点：最近的 catch 块或函数边界

**每步验证**：
- 异常传播路径上是否有未释放的资源？
- 是否使用了 RAII 保证异常安全？
- 若项目禁用异常（`-fno-exceptions`），new 失败是否返回 NULL？是否检查了 NULL？

**确认问题**：异常传播路径上存在资源泄漏，或禁用异常时未检查 new 返回值
**排除误报**：
- 使用了 RAII 包装器
- 项目使用 `-fno-exceptions` 且 new 失败会 abort

## 五、性能审查模式

### Pattern PERF-1: 循环内昂贵操作

**追踪目标**：循环体（`for`/`while`/`do-while`）内的文件 I/O、内存分配、系统调用、数据库查询等昂贵操作

**追踪路径**：
1. 起点：循环语句的起始位置
2. 路径：循环体内的所有操作
3. 终点：昂贵操作的调用位置

**每步验证**：
- 循环体内是否存在文件打开/关闭、网络请求、`malloc`/`new`、系统调用（`open`/`close`/`stat`）？
- 该操作是否可以提到循环外（结果在迭代间不变）？
- 循环的预期迭代次数是否较大（>100 次或取决于输入大小）？
- 是否存在可缓存的重复计算？

**确认问题**：昂贵操作在循环内重复执行，且可以通过提前计算、缓存或批量化移到循环外
**排除误报**：
- 操作的输入在每次迭代中不同，必须在循环内执行
- 循环迭代次数极少（常量且 <10）
- 操作已被内部缓存（如 `mmap` 后的文件访问）

### Pattern PERF-2: 不必要的数据拷贝

**追踪目标**：大对象的传值（by value）传递、不必要的 `std::string`/`std::vector` 拷贝

**追踪路径**：
1. 起点：大对象的创建或函数参数位置
2. 路径：对象在函数间传递的路径
3. 终点：对象被使用的位置

**每步验证**：
- 函数参数是否以值传递了大对象（`std::string`、`std::vector`、自定义大结构体）？
- 是否可以改为 `const &` 或移动语义（`std::move`）？
- 返回值是否触发了不必要的拷贝（编译器是否能进行 RVO/NRVO）？
- 容器操作中是否存在不必要的临时拷贝（如 `push_back(obj)` 而非 `emplace_back(...)` 或 `push_back(std::move(obj))`）？

**确认问题**：大对象被不必要地拷贝，可通过引用传递、移动语义或就地构造避免
**排除误报**：
- 对象很小（如 `int`、`double`、小于 16 字节的 POD），值传递效率更高
- 调用方确实需要一份独立拷贝（后续会修改）
- 编译器已优化掉拷贝（RVO/NRVO/copy elision）

### Pattern PERF-3: 算法复杂度不当

**追踪目标**：在频繁调用路径或大数据集上使用的高复杂度操作

**追踪路径**：
1. 起点：热点函数或频繁调用路径（从 `main` 或请求处理入口追踪）
2. 路径：函数内的循环嵌套与数据结构操作
3. 终点：疑似 O(n^2) 或更高复杂度的操作

**每步验证**：
- 嵌套循环遍历的数据集大小是否可能较大？
- 是否在循环内对 `std::vector`/`std::list` 做线性查找（O(n)），导致外层循环整体为 O(n^2)？
- 是否有更高效的数据结构可替代（如 `std::unordered_map` 替代 `std::vector` 的线性查找）？
- 排序算法是否在已部分有序的数据上使用了非自适应排序？

**确认问题**：数据规模可变且可能很大时，使用了不必要的高复杂度算法
**排除误报**：
- 数据规模有已知上限且很小（如 n ≤ 20）
- 高复杂度操作仅在初始化阶段执行一次
- 代码注释说明了选择当前算法的原因（如缓存局部性优于理论复杂度）

## 六、可维护性审查模式

### Pattern MAINT-1: 魔法数字与硬编码

**追踪目标**：代码中出现的未命名常量值、硬编码的路径/端口/超时/阈值

**追踪路径**：
1. 起点：函数体内的数字字面量或字符串字面量
2. 路径：该字面量被使用的上下文
3. 终点：判断该字面量是否有语义含义且缺少命名

**每步验证**：
- 数字字面量是否有明确语义（如 `if (retry_count > 3)` 中的 `3` 是什么含义？）？
- 是否有多处使用相同的魔法数字（不一致风险）？
- 字符串字面量中是否硬编码了文件路径、IP 地址、端口号、URL 等环境相关值？
- 超时值、缓冲区大小、重试次数等是否应提取为配置或常量？

**确认问题**：关键逻辑中使用了未命名常量，修改时需全局搜索替换，易遗漏且降低可读性
**排除误报**：
- 值为 0、1、-1 等在上下文中含义明确的通用常量
- 值已在注释中充分解释
- 值仅出现一次且上下文含义明确（如 `malloc(sizeof(int) * 2)` 中的 `2`）

### Pattern MAINT-2: 过长函数与高圈复杂度

**追踪目标**：超过合理长度或圈复杂度阈值的函数

**追踪路径**：
1. 起点：函数定义
2. 路径：函数体内的分支（if/else/switch/case）、循环、异常处理
3. 终点：函数结束

**每步验证**：
- 函数行数是否超过 100 行（建议阈值，可按项目约定调整）？
- 嵌套深度是否超过 4 层？
- 分支数（if/else/switch/case）是否过多，导致难以理解所有路径？
- 函数是否承担了多个不同的职责（违反单一职责）？

**确认问题**：函数过长或圈复杂度过高，严重影响代码可读性、可测试性和维护成本
**排除误报**：
- 函数虽长但结构清晰（如数据表初始化、协议状态机的 switch-case）
- 函数已有充分的单元测试覆盖
- 函数内有明确的注释分段且每段逻辑独立

### Pattern MAINT-3: 设计原则违反

**追踪目标**：违反 SOLID 等设计原则的模块/类结构

**追踪路径**：
1. 起点：类或模块的定义
2. 路径：类的公共接口、依赖关系、继承层次
3. 终点：违反设计原则的具体表现

**每步验证**：
- **单一职责（SRP）**：类是否承担了多个不相关的职责？是否有多个不同的修改原因？
- **开闭原则（OCP）**：新增功能是否需要修改已有代码（而非通过扩展实现）？
- **依赖倒置（DIP）**：高层模块是否直接依赖低层实现细节（而非抽象接口）？
- **接口隔离（ISP）**：接口是否过于庞大，迫使实现者实现不需要的方法？

**确认问题**：设计违反导致代码耦合度高、修改波及范围大、难以独立测试
**排除误报**：
- 项目规模小，过度设计的成本高于设计违反的代价
- 违反仅存在于内部实现，不影响对外接口
- 代码注释说明了当前设计的权衡理由

## 七、可移植性审查模式

### Pattern PORT-1: 平台特定 API 调用

**追踪目标**：无条件编译保护的平台特定 API 使用

**追踪路径**：
1. 起点：源文件中的 API 调用或头文件包含
2. 路径：检查调用点是否在 `#ifdef _WIN32`/`#ifdef __linux__` 等条件编译块内
3. 终点：确认 API 是否仅在对应平台上可用

**每步验证**：
- 是否使用了 Windows 特有 API（`CreateFile`、`WaitForSingleObject`、`HANDLE` 类型）而无 `#ifdef _WIN32` 保护？
- 是否使用了 Linux 特有 API（`epoll`、`inotify`、`signalfd`、`/proc` 文件系统）而无 `#ifdef __linux__` 保护？
- 是否使用了 POSIX 但非 C/C++ 标准的 API（`fork`、`pipe`、`mmap`）且目标平台包含非 POSIX 系统？
- 是否使用了 GCC/Clang 特有的编译器扩展（`__attribute__`、`__builtin_*`）而无编译器检测？

**确认问题**：代码中存在未被条件编译保护的平台特定 API，在目标平台上无法编译或运行
**排除误报**：
- 项目明确声明仅支持单一平台
- API 调用在条件编译块内
- 使用了跨平台抽象层（如 Boost、POSIX 兼容层）

### Pattern PORT-2: 字节序与数据类型假设

**追踪目标**：网络数据解析中对字节序的隐式假设，以及对整型大小的隐式假设

**追踪路径**：
1. 起点：网络数据接收/协议解析位置，或跨平台数据结构定义
2. 路径：数据从原始字节到结构化类型的转换过程
3. 终点：多字节整数值被使用的位置

**每步验证**：
- 多字节整数从网络字节流中解析时是否使用了 `ntohl`/`ntohs` 等字节序转换？
- 是否通过 `memcpy` 或指针强转将字节流直接映射到结构体（可能受对齐和字节序影响）？
- 是否假设 `int` 为 32 位、`long` 为 64 位或 `size_t` 为特定大小？
- 结构体是否使用了 `#pragma pack` 或 `__attribute__((packed))` 确保跨平台布局一致？

**确认问题**：代码隐式假设了特定字节序或数据类型大小，在不同架构/平台上可能产生错误结果
**排除误报**：
- 已使用标准字节序转换函数（`htonl`/`ntohl` 系列）
- 使用了固定宽度类型（`uint32_t`/`int64_t`）代替平台相关类型
- 项目仅运行在已知的单一架构上
